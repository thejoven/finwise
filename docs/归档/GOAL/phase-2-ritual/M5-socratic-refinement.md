# M5 · 五轮追问对话

> Phase 2 · W9-W11 · 3 周 · LLM 对话核心

---

## 上下文

M5 是 Phase 2 的入口, 也是产品**第一次真正和用户对话**。

它是 "AI 出题, 你做选择" 范式的实现——LLM 不替用户思考, 它出题让用户在选项中暴露认知。

完成后, 当用户在 A1 看到 "AI 找你聊一条信号" 时, 点开就进入这套追问流程。

---

## 前置依赖

- ✅ Phase 1 全部完成(M1-M4)
- ✅ 后端有累积的真实信号数据(W8 自己用过的)

---

## 目标

### 后端
- Mastra Socratic Agent(出题 Agent)
- Refinement Workflow(五轮对话编排)
- SSE 流式回传 LLM 输出到客户端
- 答题结果写 events 表(refinement.answered)

### 客户端
- 进入追问页(每轮一张卡)
- 显示选项 / 多选 / 排序 / 开放回答 4 种题型
- SSE 接收 LLM 流式问题
- 提交答案 → 进下一轮
- 五轮结束 → 决定是否进 M6 评估

### 数据模型
新增物化视图 `refinement_sessions` 跟踪每条信号的对话进度。

## 任务列表(高层)

### Task 5.1 · Socratic Agent prompt 设计

参考 `产品文档/02_第二层_模式识别与推演训练.md`。

关键约束:
- 出题不给答案
- 选项必须有干扰项
- 题目质量取决于干扰项质量
- 不超过 5 轮, 不超过 4 个选项
- 题型: 单选 / 多选 / 排序 / 开放(用于复杂场景)

### Task 5.2 · Mastra Refinement Workflow

5 步 workflow, 每步:
1. Socratic Agent 出一道题
2. 推送给客户端(SSE)
3. 等用户答(可能离线, 用 NATS 队列)
4. 答题写 events 表
5. 决定下一题方向(根据答案动态)

**已知坑**:
- 第 5 步不要让 LLM 完全自由, 限制下一题的范围(锚定、推演、估值三类)
- 每轮答完都存 state, 用户可以中断后续上

### Task 5.3 · SSE 流式接口

`POST /v1/refinement/:signal_id/next` → SSE 流

事件:
- `question` — 一道完整题目
- `option` — 选项
- `complete` — 题目结束
- `error` — 错误

### Task 5.4 · 客户端追问页

`app/signal/[id]/refine.tsx` 实现:
- 顶部进度(I / V, II / V, ...)
- 卡片样式问题(报刊感)
- 选项列表(TapEffect)
- 提交按钮
- SSE 流式接收下一题

### Task 5.5 · 答完五轮的决策

五轮全过 → 系统决定是否进 M6 评估(四道门)。
答错率高 → 训练记录, 不进评估。

## 验收标准

- [ ] 至少跑通 1 次完整 5 轮对话
- [ ] 每轮 SSE 流式显示流畅, 没有卡顿
- [ ] 选项有真正的干扰项(不是显然错的)
- [ ] 答题数据完整写入 events 表
- [ ] 离线答题再联网, 继续上一轮
- [ ] 视觉报刊感, 每张卡像一封信

---

## 反模式特别警告

- ❌ 不要做成 chatbot 形态(气泡 + 头像 + "AI 正在输入")
- ❌ 不要让 LLM 给"标准答案"
- ❌ 不要在用户答错时弹"再试一次"
- ❌ 不要"鼓励性反馈"("答得不错!")
- ❌ 不要用 Material Card 包问题

正确形态:**像一份关于你认知的报刊问卷**, 每题独立, 像一封正式信件。

---

## 已知坑

1. **Socratic Agent 的 prompt 反复迭代 5 次以上才能稳定**, 不要一次写完就用
2. **题型 schema 严格用 zod 限制**, LLM 不能自由发挥
3. **SSE 在 RN 上要用 fetch-event-source**, 不是 EventSource
4. **答题中断恢复**是必须功能(用户可能在第 3 题被电话打断)
5. **5 轮是上限, 不是目标** — 第 3 题就明显答对了, 可以提前结束

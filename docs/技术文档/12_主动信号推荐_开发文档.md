## § 0 · 一句话与一条红线

**一句话**:把订阅 firehose 当原料,用你**已经留下的行为轨迹**算出"对你而言什么是 alpha",在你**已经会看的地方**(订阅 feed / 归档 / 承诺复盘)安静地标出极少数值得注意的条目,每条都附一句"为什么是给你的"。

**一条红线 —— 这是本功能存亡的前提**:
本产品的全部价值是**降噪**——四道门、降噪页、沉默归档([产品哲学](../产品文档/06_产品哲学.md)、[gate.evaluated 沉默原则](../../server/internal/module/gate/service.go))。一个"给你推荐信号"的功能,如果做成推送流 / 发现页 / 红点 feed,就会**精确地变成产品要消灭的那台 FOMO 噪音机器**。

所以本功能不是"发现引擎",是**策展(curation)引擎**:

- **宁可不推**。清不过线就沉默。硬上限,不是软建议。
- **不主动推送**。推荐是被你**撞见**(在你已经会打开的界面里),不是被 push 出来打断你。默认无通知。
- **每条必带"为你"的理由**,且理由必须落在你自己的历史上("你在 X 上签过承诺;这条与你写的退出条件相关")。没有黑箱"猜你喜欢"。
- **可否、可学**。一键"不相关"即是训练信号,回灌画像。

把这条红线违反了,这个功能不该上线。下面所有设计都服从它。

## § 2 · 用户 alpha 画像 —— "训练久了"到底学了什么

用户口中的"训练久了 / 根据你的认知",落到工程上 = 一个**每用户的 alpha 画像**,完全由**已经存在的行为轨迹**派生。我们不需要让用户填问卷,产品早已在记录:

| 画像维度 | 信号含义 | 数据来源 (均已落库) |
|---|---|---|
| **领域亲和** tag/category affinity | 你真正下功夫的赛道 | `signals.inference_tags`、被你"转信号"的 `tweets.tags/category`、完成五轮的 `refinement_sessions` |
| **确信形态** | 你在什么样的信号上**真的过得了门** | `gate_evaluations.passed / failed_gate / gates_detail` |
| **已表态偏好**(最强) | 你**真金白银签下**的命题 | `commitments.thesis.{asset, action, reasons_for_future_self}` |
| **自知弱项** | 你复盘时自己认的短板 | `retrospects.focus_dim / focus_text`(录入速度/推演深度/决策速度/持仓耐心…) |
| **认知体感** 四维 | 专注 / 深度 / 广度 / 执行 | [`/v1/attention/summary`](../../server/internal/module/attention) 的四维评分 |
| **透镜偏好** | 你惯用的研究视角 | [lens.ts L1–L10](../../mastra/src/agents/lens.ts) 在你历史会话里的命中 |
| **持仓上下文** | 你现在押着什么、退出条件是什么 | `commitments`(active) / `holdings` |
| **客观地面真值**(P4+) | 你看对过什么——标的真涨了吗 | [13 标的追踪](13_标的追踪_开发文档.md) 的 `signal_assets` 收益 |

**关键洞察**:前 7 维全是**自陈/行为**信号(你说你在乎、你点了、你签了),只有最后一维是**客观结果**。没有结果维度,画像只会越来越像"你过去的样子",而不是"你看得准的样子"。所以 [13](13_标的追踪_开发文档.md) 不是锦上添花,是让"alpha"这个词名副其实的前提——本文 P4 必须接上。

**实现取舍(混合)**:

- **Go 侧 `user_alpha_profile` 表**(物化、可快取):承载可量化、需要快速预筛的部分(tag/category 权重向量、确信形态统计、活跃持仓快照)。由一个 builder 周期性重算(派生投影,**不写 events**——与 [distillation/subscription 先例](11_推文订阅_开发计划.md#§-0-架构修订-相对规格文档-2-的一处简化)一致)。
- **Mastra working memory**(已有,每 `resourceId` 的 markdown 画像,见 [memory 架构](../../mastra/src/memory/agent-memory.ts)):承载需要语言理解的细腻部分(决策模式、透镜偏好、风险偏好),由策展 agent 读写。

两者分工 = "粗排在 Go,精排在 Mastra",与订阅"派发在 Go、分类在 Mastra"同构。

## § 3 · 候选池与收敛漏斗

不另起采集——**复用订阅 firehose**。候选 = 你订阅源里 **未读 + 高 relevance** 的 `tweets`(已被 `tweet-classifier` 打好 `tags/category/relevance/summary`)。

四级漏斗,逐级变贵、变窄:

```
firehose (tweets, 已分类)
  │  ① relevance 阈值 + 去重(已读/已转信号/近 N 天已推过)        ← Go, 廉价
  ▼
候选 (per user, 数十条/日)
  │  ② alpha 画像粗排:tag/category 亲和 × 持仓相关度 × 时新性     ← Go, user_alpha_profile
  ▼
Top-K (≤ 10~20)
  │  ③ Mastra 策展 agent:给定[画像摘要 + 活跃命题 + K 条候选],
  │     判"这对**这个**用户是不是 alpha",产出排序 + 每条一句"为你"理由 ← Mastra, 贵
  ▼
精选 (≤ 配额)
  │  ④ 硬配额 + 沉默闸:不过线就丢弃,不凑数                        ← Go
  ▼
recommendations 表 (待呈现)
```

- ① ② 在 Go,沿用订阅 dispatcher 的 `SELECT … FOR UPDATE SKIP LOCKED` 取批模式。
- ③ 新增 Mastra `recommender`(策展)agent,与现有六个无状态端点同形(见 [http.ts](../../mastra/src/server/http.ts));注入 [market-context](../../mastra/src/agents/market-context.ts) 与画像摘要。
- ④ **配额是产品红线的执行点**:默认 ≤3 条/日全局;承诺相关情报另算(见 §4)。清不过线→当天沉默。

## § 4 · 推荐出现在哪 —— 嵌入式,而非一个新 tab

这是全功能**最重要的产品决定**。推荐绝不做成第五个 tab 或推送流。它出现在用户**自己说有参考价值的两个地方**,以及它天然属于的订阅页:

| 位置 | 形态 | 为什么噪音最低 / 价值最高 | 涉及现有界面 |
|---|---|---|---|
| **① 承诺 / 复盘** (最高价值, 先做) | "与这条命题相关的新进展" —— 尤其**反证** | 绑定在你**已经签下**的命题上,几乎不可能是无关噪音;反证正是复盘最缺的 | [`/commitment/[id]`](../../mobile/app/commitment), [`/retrospect/[id]`](../../mobile/app/retrospect) |
| **② 归档** | "你当时放下的想法,订阅里出现了相关新进展" | 被四道门否掉的命题,此刻是否值得**重提**——高参考、低频 | [`ArchiveView`](../../mobile/src/features/archive/ArchiveView.tsx) + [归档分析师对话](../../mobile/app/archive/chat) |
| **③ 订阅 feed** | "为你标记":在 ≤N 条上钉一个克制的标记 + 一句理由 | feed 本就要看,这只是**精排叠加**,不新增打断 | [订阅 feed](../../mobile/src/features/subscription) |
| **④ 统计**(可选, 后做) | 每周"本周值得注意"数字摘要,**默认 opt-in** | 报刊式克制,周频,不打断 | [`AttentionScreen`](../../mobile/src/features/attention/AttentionScreen.tsx) |

文案口径遵循 [产品文案 YC PM 标准](../产品文档):中性、直接说动作("查看相关进展"),禁挑衅 / 抖机灵 microcopy。

**人始终在环内**:推荐**不自动建信号**。它只是"建议你看一眼",转不转信号、过不过门,仍是你手动触发——与"四道门手动触发""对话不改判"同一条产品哲学。

## § 5 · 数据模型

派生投影,**不写 events**(同 §2 先例)。两张新表 + 复用订阅/承诺。

**`user_alpha_profile`** —— 每用户画像快照(物化,可重算)

```sql
CREATE TABLE user_alpha_profile (
    user_id          uuid PRIMARY KEY,
    tag_affinity     jsonb NOT NULL DEFAULT '{}',   -- {"AI芯片": 0.8, "美债": 0.5, …} 加权
    category_affinity jsonb NOT NULL DEFAULT '{}',  -- 7 类 category 权重
    conviction_shape jsonb NOT NULL DEFAULT '{}',   -- 各门通过/否决统计、典型失败门
    lens_preference  jsonb,                          -- L1–L10 命中分布(可由 Mastra 回写)
    active_theses    jsonb NOT NULL DEFAULT '[]',   -- 活跃命题快照 [{asset, action, exit_conditions}]
    built_from_until timestamptz,                    -- 画像截至哪个时间点的行为
    sample_size      int NOT NULL DEFAULT 0,         -- 行为样本量(冷启动判据)
    updated_at       timestamptz NOT NULL DEFAULT now()
);
```

**`recommendations`** —— 多态呈现位(沿用订阅多态先例:`context_type + target_ref`)

```sql
CREATE TABLE recommendations (
    id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      uuid NOT NULL,
    source_type  text NOT NULL DEFAULT 'tweet',     -- 预留 telegram/rss
    source_id    text NOT NULL,                      -- → tweets.id
    score        real NOT NULL,                      -- 粗排+精排融合分
    rationale    text NOT NULL,                      -- 策展 agent 产出的"为你"一句话
    context_type text NOT NULL,                       -- 'feed' | 'commitment' | 'archive' | 'digest'
    target_ref   uuid,                                -- commitment_id / evaluation_id(feed 为 null)
    status       text NOT NULL DEFAULT 'pending'
                 CHECK (status IN ('pending','surfaced','dismissed','promoted','expired')),
    model        text,
    created_at   timestamptz NOT NULL DEFAULT now(),
    surfaced_at  timestamptz,
    acted_at     timestamptz,
    UNIQUE (user_id, source_id, context_type, target_ref)   -- 同位幂等,不重复推
);
```

- `status='dismissed'`(用户点"不相关")= 负反馈,builder 下次重算画像时降权该 tag/来源。
- `status='promoted'`(用户真的转了信号)= 强正反馈。
- **不删订阅表 / 不动承诺表**;承诺相关情报通过 `context_type='commitment', target_ref=commitment_id` 关联。

## § 6 · 各层改动

**Go**(`internal/module/recommend/`,风格照 [subscription](../../server/internal/module/subscription)):

- `profile_builder.go` —— 周期重算 `user_alpha_profile`(读 signals/gate/commitments/retrospects/attention 派生)。
- `builder.go` —— §3 漏斗:取候选 → 粗排 → 调 Mastra 策展 → 配额 → 写 `recommendations`。worker 化,与订阅 poller 同形(cron tick + SKIP LOCKED)。
- `handler.go` —— 读端点:
  - `GET /v1/recommendations?context=feed` —— feed 精排标记
  - `GET /v1/commitments/:id/related` —— 该承诺的相关情报(P1 入口)
  - `GET /v1/gate/evaluations/:id/related` —— 该归档命题的相关进展
  - `POST /v1/recommendations/:id/dismiss` / `…/seen` —— 反馈
- 接线 `main.go` + `config.go`(配额、cadence、relevance 阈值均 env 可调)。

**Mastra**(`src/agents/recommender.ts` + `/recommend` 端点):

- 无状态精排端点(同 [consensus-check 等六端点](../../mastra/src/server/http.ts)),入参 `{画像摘要, 活跃命题[], 候选[]}`,出参 `[{source_id, keep, score, rationale}]`。
- 注入 [market-context](../../mastra/src/agents/market-context.ts);提示词强调"对**这个**用户是否 alpha + 反证优先",并明确**默认丢弃**(宁缺毋滥,呼应红线)。
- 复用每用户 `resourceId` working memory 读画像、回写透镜偏好。

**Mobile**(`src/features/recommend/`):

- feed 精排:在 [订阅 feed](../../mobile/src/features/subscription) 行上加克制的"为你标记"+ 理由展开 + "不相关"。
- 承诺/复盘:在 [`/commitment/[id]`](../../mobile/app/commitment)、[`/retrospect/[id]`](../../mobile/app/retrospect) 加"相关进展"模块卡(空态即不渲染,**不留空槽**)。
- 归档:在 [`ArchiveView`](../../mobile/src/features/archive/ArchiveView.tsx) 卡片上加"相关新进展"角标 → 进对话。
- API 客户端照 [`core/api/gate.ts`](../../mobile/src/core/api/gate.ts) 风格新增 `core/api/recommend.ts`。

## § 7 · 分期 —— "目前阶段先做哪一刀"

按**价值高 / 噪音风险低**优先,而非按技术顺序。每期独立可上线、可验证。

| 期 | 名称 | 范围 | 为什么这个顺序 | DoD |
|---|---|---|---|---|
| **P0** | 画像底座 | `user_alpha_profile` builder,从既有数据派生;**无 UI** | "训练"的物质基础;先肉眼验证画像合理 | 抽 3~5 个真实用户画像人工核对成立 |
| **P1** | 持仓相关情报 | 仅对**活跃承诺**做策展,出现在承诺/复盘页;每命题 ≤N 条 | 绑定已签命题→几乎不可能是噪音,价值最直接,最稳妥的第一刀 | 205 上:某承诺能稳定召出相关 tweet + 反证,误召人工可接受 |
| **P2** | feed 精排 + 为你标记 | 订阅 feed 排序 + ≤3/日钉标 + 理由 + "不相关"反馈闭环 | feed 已在看,叠加不新增打断;反馈开始喂画像 | 配额硬生效;dismiss 能降权;沉默日真的沉默 |
| **P3** | 归档复盘联动 | 归档命题的"相关新进展",可重提 | 让被否的想法有第二次机会,纯增量 | 归档卡角标→对话链路通 |
| **P4** | 周报数字 + 结果回灌 | opt-in 周摘要;接入 [13](13_标的追踪_开发文档.md) 的标的收益作为**地面真值**校准画像 | 此时画像才从"像你过去"进化到"像你看得准";依赖 13 先到 P2+ | 画像 conviction_shape 纳入真实收益;A/B 可见召回质量提升 |

**当前阶段建议**:先 **P0 → P1**。P1 是整个功能的试金石——如果连"绑死在你已签命题上的情报"都做不到既相关又克制,就不该往 feed(P2)扩。

## § 8 · 不变量、冷启动、隐私、失败模式

- **沉默不变量**:`recommendations` 可以为空。任何 UI 位空态即不渲染(不留"暂无推荐"空槽——空槽本身是噪音)。
- **配额不变量**:全局日配额是**硬上限**,在 §3 ④ 强制;承诺相关情报单独小配额。越线即丢。
- **冷启动**:`sample_size` 低于阈值时,builder 标记画像"未成熟",builder 直接跳过该用户或仅退化为纯 relevance 排序,且不在 feed 钉标——呼应用户原话"用久 / 训练久了才有"。诚实承认"还不懂你"好过乱推。
- **隐私 / 不跨用户**:画像与 working memory 均按 `resourceId` 严格隔离(沿用 [memory 架构](../../mastra/src/memory/agent-memory.ts) 的 `scope:'resource'`);`tweets` 全局共享但 `recommendations`/画像 per-user。
- **不自动行动**:推荐永不自动建信号 / 自动过门;人手动触发不变(同 [手动四道门](03_数据模型与事件溯源_大纲.md) 哲学)。
- **失败模式**:Mastra 不可用→粗排照常出,精排降级为"仅承诺相关 + relevance 排序",不阻塞;策展返回全丢→当天沉默(正确行为,不是 bug)。

## § 9 · 开放决策(执行前需拍板)

1. **画像放 Go 表 vs. 全靠 Mastra working memory** —— 本文建议混合(粗排量化进 Go 表,精排细腻留 Mastra)。若想更轻可先纯 Mastra,但预筛会变慢。
2. **builder cadence** —— 日 cron vs. 每轮 poll 后增量。建议日 cron 起步(够用、省 LLM)。
3. **配额默认值** —— feed ≤3/日、每承诺 ≤2/周?需产品定标后写进 config。
4. **P4 依赖** —— 结果回灌依赖 [13](13_标的追踪_开发文档.md) 至少到 P2;两文档分期需对齐排程。

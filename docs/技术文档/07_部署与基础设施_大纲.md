## 章节结构

### § 1. 本地开发环境

Docker Compose 一键起所有依赖:

```yaml
services:
  postgres:        # PostgreSQL 16 + pgvector 扩展
  redis:           # Redis 7
  nats:            # NATS JetStream
  go-api:          # 本地构建的 Go 服务
  mastra:          # 本地 Node.js 服务
```

包含:

- 一份完整 `docker-compose.dev.yml`
- 环境变量样例 `.env.example`
- 数据库初始化脚本(自动跑 migration + 注入测试用户)
- 启动命令清单(make 目标)

### § 2. Phase 1 生产部署形态

🟢 Phase 1 不上 K8s,直接单机 Docker Compose:

- VPS 配置建议:4 核 8G(够你一个人用,LLM 调用是远程)
- TLS:Caddy 反向代理,自动证书
- 备份:每日 pg_dump → 对象存储 (R2 / S3)
- 监控:简单的 healthcheck + uptime ping

成本估算:VPS 月 ~$20 + LLM API ~$30 + 对象存储 ~$1 = **~$50 / 月**(单用户)。

### § 3. 长期生产形态(Phase 3+ ⚠️)

如果将来要支持多用户,可能演进到:

- K8s 或 Fly.io / Railway
- 数据库托管(Supabase / Neon)
- LLM 调用走 LiteLLM proxy

这部分只画大方向,不细化。

### § 4. 密钥管理

- 本地:`.env` 文件,不入 git
- 生产:Docker secrets 或 systemd EnvironmentFile
- LLM API key 轮换策略:手动,Phase 1 不自动
- JWT secret:首次部署生成,后续不轮换

### § 5. 日志策略

- 所有服务输出结构化 JSON 日志到 stdout
- 收集:用 `vector` 或 `fluentbit`,落到本地文件 + R2
- 检索:Phase 1 直接 grep,Phase 2 上 Loki

关键字段:

- `trace_id`(贯穿 Flutter → Go → Mastra)
- `user_id`
- `event_type`
- `latency_ms`

### § 6. 可观测性

**指标(Prometheus)**:

- 业务指标:`signals_captured_total`、`gates_passed_total`、`commitments_signed_total`
- 系统指标:HTTP 延迟、Go GC、数据库连接池

**追踪(OpenTelemetry)**:

- Flutter / Go / Mastra 都接 OTel SDK
- 后端 collector 用 `tempo` 或 `jaeger`
- 关键追踪:一条信号从录入到推演完成的完整 span

**仪表盘**:

- Grafana 一个 dashboard,Phase 1 只展示业务指标(不上系统指标)

### § 7. 健康检查

- `/health` — 简单的 200(配 Caddy 用)
- `/ready` — 检查数据库、Redis、NATS 连通(配 K8s readiness)
- `/v1/ops/diagnose` — 深度诊断,管理员触发

### § 8. 数据备份与恢复

**备份**:

- 每日 pg_dump,加密上传 R2
- 保留 30 天

**恢复**:

- 文档化恢复流程(从 dump 到服务可用)
- 定期演练(Phase 2 起每季度一次)

呼应产品哲学:用户的事件流是他们人生切片的备份,我们对它的可用性必须负责。

### § 9. CI/CD

🟢 Phase 1 最简:

- GitHub Actions 跑测试 + 构建 Docker 镜像(后端)
- **客户端用 EAS Build** 出 iOS / Android 构建产物
- 主分支合并后, 后端镜像推到 GHCR, 客户端通过 EAS 发到 TestFlight
- 手动 SSH 上 VPS 拉镜像、`docker compose up -d`

**Expo / EAS 配置要点**:

- `eas.json` 配置 development / preview / production 三个 profile
- Development build 用 Expo Go 模式调试
- Preview build 出 TestFlight / Internal Testing
- Production build 用于 App Store 提交
- OTA 更新通过 `eas update` 推送(JS 改动免审核更新)

OTA 关键约束:**只推 JS 改动, Native 改动必须重新提交审核**。
所以慎用 Native 模块, 用 Expo 提供的优先。

Phase 2 起接 ArgoCD 或类似。

### § 10. 安全基线

- HTTPS 全程
- JWT 短 TTL (15min) + Refresh Token
- 限流:Redis 计数器,登录失败 5 次锁 15 分钟
- SQL 注入防护:Ent ORM 已经保护
- XSS 防护:Flutter 没有 HTML 渲染,N/A

---

## 关键决策预告

1. **Phase 1 单机 Docker Compose,不上 K8s** — 一个人维护得过来
2. **Caddy 而不是 Nginx** — 自动 TLS,配置简单
3. **PostgreSQL 单库** — 不引入额外数据库
4. **不接 ELK 全家桶** — Phase 1 grep 足够
5. **手动部署可以,自动化不必** — 一周一次发布的频率,手动反而稳

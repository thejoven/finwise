## § 0 · 两个已拍板的决策

开工前先把三个会改变架构的产品决策钉死, 后面所有设计都从这几条推出来:

| 决策 | 选择 | 影响 |
|---|---|---|
| **导航位置** | **底部独立 tab「订阅」, 位于财知左侧** | 恢复四 tab (`订阅 · 财知 · 统计 · 我`), 顶替旧「报纸」的位置 (报纸当年就是"财知左侧的新栏目"); 启动仍落在财知 (`initialRouteName: "caizhi"` 不变)。未读数做成 tab 上的小红点。 |
| **MVP 范围** | **阅读 + AI 标签/总结 + 已读, 且含「转为信号」桥接** | 详情页加「转为信号」入口, 一键把推文落进现有信号管道 (进而可走推演 / 投决会)。把订阅和产品核心闭环打通。 |
| **订阅的可扩展性** | **管理页类型优先 + 订阅关系表 v1 起多态** | 未来订阅 Telegram 频道 / 新闻源 (RSS) 不动骨架: 每类型一张源表 + 一个采集器, `subscriptions(source_type, source_id)` 通吃 (§4.2); 管理页 v1 即展示类型选择, 未上线的标「规划中」(§8.3)。 |

「订阅」不是第二个信号源的平替, 它是**外部输入的过滤层**: 替你盯着该盯的人, 把噪音挡掉, 偶尔捞出一条值得变成「信号」的东西。这与 [产品哲学](../产品文档/06_产品哲学.md) 一致——离信号最近、离仓位最远的时刻, 往往就藏在你关注的人的一条推文里。

## § 1 · 功能范围与用户故事

### 1.1 用户故事

1. **订阅**: 我输入一个 `@handle` (如 `@solana`), 它成为我的订阅。我能看订阅列表、随时取消。
2. **采集**: 系统定时去 twtapi 拉这些账号的新推文, 我不用手动刷新。
3. **阅读**: 「订阅」tab 里是一条按时间倒序的推文流, 每条带 **AI 标签** + **一句话总结**, 未读的有标记。
4. **已读**: 点开一条 = 标记已读; 也能「全部已读」。tab 上的未读小红点清零, 我心里有底「订阅都读完了」。
5. **转信号**: 看到一条值得下注的, 点「转为信号」, 它落进信号管道, 后续能走推演 / 投决会。

### 1.2 明确不做 (MVP 边界)

- ❌ 不做发推 / 点赞 / 转发等**写操作** (twtapi 也只读)。
- ❌ 不做全文搜索推文、不做关键词订阅 (只订阅**账号**; 关键词订阅留作后续, 见 §9)。
- ❌ 不做实时推送通知 (Phase 后续再接 [PushNotification], MVP 靠打开 app 拉取)。
- ❌ 不做多媒体的离线缓存 (图片/视频只存 URL, 客户端按需加载)。
- ❌ 已读状态**不做事件溯源** (高频、低价值, 走普通表 upsert, 见 §4.4)。

## § 2 · 整体架构与数据流

> ⚠️ **架构修订 (2026-06-10, 执行时)**: 分类链路从下图的「事件 → outbox → iii 队列」**简化为「poller 落库 (pending) + Go 派发器同步调 Mastra /tweet-classify + 直接回写」**。
> 理由与新图见 [11 · 开发计划 § 0](11_推文订阅_开发计划.md): 推文是全局系统数据, 不是 per-user 领域事件, 不该进 events 表 (distillation 模块同先例); `classify_status=pending` + 重试本身就是持久化队列。
> 下图与 §5.3/§7.2 的 iii 队列方案保留作为原设计记录, **以 11 § 0 为准**。

完全复用现有信号管线的形状 (capture → 事件 → outbox → iii 队列 → Mastra → 写回 → 物化视图), 这样不引入新范式。见 [ADR 0004 · iii over NATS](../adr/0004-iii-over-nats.md)。

```

                         │  每 N 分钟: 对每个被订阅账号 GET twtapi UserTweets(user_id, cursor) │
                         │  按 tweet_id 去重 → 只插新推文 → 更新 high-water / last_polled_at  │
                         └───────────────────────────────┬──────────────────────────────────┘
                                                         │ 每条新推文 (一个事务内):
                                                         │   tweets 行 + tweet.captured 事件 + outbox 行
                                                         ▼
   twtapi.com  ◀──HTTP──  infra/twtapi.Client      outbox worker ──POST──▶ Mastra HTTP shim
   (X 数据代理)            (UsernameToUserId /        (已存在)               /v1/events/tweet-captured
                          UserTweets / Search /                                    │ enqueue
                          TweetDetail)                                             ▼
                                                                          iii 队列 q:tweet-classify
                                                                                   │
                                                                                   ▼
                                                                       runTweetClassifier (DeepSeek)
                                                                          → { tags[], summary, ... }
                                                                                   │ POST 回
                                                                                   ▼
                                                          Go  POST /v1/internal/tweets/classify-result
                                                          → 写回 tweets.tags / summary / classify_status=done
                                                          → (可选) emit tweet.classified

   移动端「订阅」tab  ──GET /v1/tweets?filter=unread──▶  Go subscription 模块  ──读──▶  tweets / tweet_reads
                     ──POST /v1/tweets/:id/read──────▶  (handler/service/repo)
                     ──POST /v1/tweets/:id/promote───▶  复用 signal.Capture → signal.captured (桥接)
```

三个关键点:

1. **采集 (Go poller)** 和 **分类 (Mastra)** 解耦: poller 只负责「快进快出」把推文落库, 分类是异步的, 不阻塞采集。这与信号 `signal.captured → q:signal-inference → runSignalInference` 同构 (见 [`mastra/src/iii/worker.ts`](../../mastra/src/iii/worker.ts))。
2. **推文存储是共享的, 订阅/已读是 per-user**: 多个用户订阅同一账号, 推文只采一份、只分类一次 (省 twtapi 调用 + DeepSeek 调用), 见 §4。
3. **转信号是「零新管道」**: 直接复用 [`server/internal/module/signal`](../../server/internal/module/signal) 的 `Capture`, 用推文文本当 `raw_text`, 不重新发明轮子, 见 §5.5。

## § 3 · 第三方接入 · twtapi.com

文档: https://www.twtapi.com/zh/docs/ 。它是 X 官方 API 的代理 (免爬虫直连 x.com API)。

### 3.1 鉴权 / Base URL

- **Base URL**: `https://api.twtapi.com/api/v1/twitter/`
- **鉴权**: 请求头 `X-API-Key: <KEY>` (在 twtapi 后台获取)。可选 `X-Lang: zh`。
- **统一返回信封**: `{ code, msg, data }`。`code` 是**业务状态码** (不一定等于 HTTP 状态码), `200` 为成功, 读 `data`。
  - ⚠️ 文档里 Trends/Search 的示例又写成 `{ success, data }` —— **信封字段不一致, 这是 §8 头号风险, Phase 0 必须实测确认**。

错误码 (业务 `code`):

| code | 含义 | 处理 |
|---|---|---|
| 200 | 成功 | 读 `data` |
| 400 | 参数错误 | 检查必填/取值 |
| 401 | 未授权 (key 无效) | 检查 `X-API-Key` |
| **402** | **余额不足 / 调用次数用完** | **告警 + 暂停 poller, 不要空转烧重试** |
| 403 | 无权限 / 账户受限 | 查账户状态 |
| 404 | 资源不存在 | 账号被删/改名 → 标记订阅失效 |
| **429** | **触发限流 (秒/小时/月)** | **退避重试 (backoff)** |
| 500/502/503 | 服务端异常 | 稍后重试 |

### 3.2 用到的 endpoints

| 用途 | Method + Path | 关键参数 | 备注 |
|---|---|---|---|
| 用户名 → 用户ID | `GET /UsernameToUserId` | `username` (不含 @) | 解析一次, 永久缓存到 `twitter_accounts.rest_id` |
| 账号资料 | `GET /UserResultByScreenName` | `username` | 取头像/昵称/简介, 订阅时拉一次 |
| **用户推文 (主力)** | `GET /UserTweets` | `user_id` (Rest ID, 必填), `cursor` (翻页, 可选) | **采集主力**。最新在前, 翻页靠 `cursor` |
| 用户推文+回复 | `GET /UserTweetsReplies` | `user_id`, `cursor` | 想连回复一起收时用 (MVP 默认不收回复) |
| 搜索 (备选) | `GET /Search` | `q`, `type` (Top/Latest/...), `count` | `q=from:handle&type=Latest` 可绕过 user_id; 见 §3.4 |
| 单条推文/线程 | `GET /TweetDetail` | `tweet_id` (Rest ID) | 详情页展开整条线程时用 |

调用示例 (curl):

```bash
# 1. 解析 handle → rest id (每个账号一次, 缓存)
curl -X GET "https://api.twtapi.com/api/v1/twitter/UsernameToUserId?username=solana" \
  -H "X-API-Key: $TWTAPI_API_KEY"

# 2. 拉该账号最新推文 (poller 主循环)
curl -X GET "https://api.twtapi.com/api/v1/twitter/UserTweets?user_id=951329744804392960" \
  -H "X-API-Key: $TWTAPI_API_KEY"
```

### 3.3 响应结构 · ✅ 已实测固化 (2026-06-09)

P0 spike 已用真实 key 跑通五个端点, 原始样本存 [`docs/api/samples/twtapi/`](../api/samples/twtapi)。关键发现:

- **信封不统一**: `UsernameToUserId` 直接返 `{id, id_str}` (无信封); `UserTweets`/`Search`/`TweetDetail`/`UserResultByScreenName` 直接返 `{data: ...}` (也**不是** `{code,msg,data}`)。文档里的 `{code,msg,data}` / `{success,data}` 都不准, **以实测为准**。HTTP 状态码正常 (成功 200)。
- **`UserTweets` 透传 x.com GraphQL 原始时间线** (单页 ≈180KB, 深度嵌套), **不是**文档示例那种 `{tweets:[{id,text}]}`。
- **`Search` 额外带一个 twtapi 自加的 `_normalized` 块** (铺平的 `tweets[]` + `next_cursor`/`previous_cursor`), 比 UserTweets 好解析得多 —— 但**只有 Search 有 `_normalized`**, UserTweets/TweetDetail 没有。
- **三个端点的「单条推文对象」结构完全一致** (记为 `<TWEET>`), 所以**解析器只写一份, 三处复用**。

**单条推文 `<TWEET>` 字段映射** (相对 `<TWEET>` 根):

| 含义 | 路径 | 类型 / 示例 |
|---|---|---|
| 推文 id | `rest_id` | `"2064238810582438366"` |
| 正文 | `legacy.full_text` | string |
| 发布时间 | `legacy.created_at` | `"Tue Jun 09 06:50:58 +0000 2026"` (Twitter ruby 格式, Go: `time.Parse("Mon Jan 02 15:04:05 -0700 2006", v)`) |
| 语言 | `legacy.lang` | `"en"` |
| 会话 id | `legacy.conversation_id_str` | string |
| 回复目标 | `legacy.in_reply_to_status_id_str` | string \| null |
| 是否引用 | `legacy.is_quote_status` | bool |
| 引用原推 id | `legacy.quoted_status_id_str` | string |
| 引用原推对象 | `quoted_tweet_results.result` | 嵌套同 `<TWEET>` |
| 点赞 / 转发 / 回复 / 引用 / 收藏 | `legacy.{favorite,retweet,reply,quote,bookmark}_count` | int |
| 浏览数 | `view_count_info.count` | `"20679090"` (**string**) |
| 媒体 | `legacy.extended_entities.media[]` (兜底 `legacy.entities.media[]`) | 见下 |
| 作者 id | `core.user_results.result.rest_id` | `"44196397"` |
| 作者昵称 | `core.user_results.result.core.name` | `"Elon Musk"` |
| 作者 handle | `core.user_results.result.core.screen_name` | `"elonmusk"` |
| 作者头像 | `core.user_results.result.avatar.image_url` | url |

**媒体项** (`legacy.extended_entities.media[]` 每项): `type` (`photo`/`video`/`animated_gif`) · `media_url_https` (photo 直接用, video 是封面图) · 视频源 `video_info.variants[]` (`{content_type, url}`, 取 `video/mp4` 最高码率)。

**从各端点取 `<TWEET>` 列表的容器路径** (这是唯一不同的地方):

| 端点 | 推文列表路径 | 翻页游标 |
|---|---|---|
| **UserTweets** | `data.user_result_by_rest_id.result.profile_timeline_v2.timeline.instructions[].entries[]` → 过滤 `content.__typename == "TimelineTimelineItem"` → 取 `content.content.tweet_results.result` | cursor 项: `content.__typename == "TimelineTimelineCursor"` 且 `content.cursor_type == "Bottom"` 的 `content.value` |
| **Search** | `_normalized.tweets[].result` (twtapi 已铺平) | `_normalized.next_cursor` |
| **TweetDetail** | `data.tweet_result.result` (单条) | — |

**RT 检测**: `legacy.full_text` 以 `RT @` 开头, 或存在 `legacy.retweeted_status_result` (本次 elonmusk 样本里转推以 `RT @` 文本呈现, 19 条里 10 条是转推) → 置 `is_retweet`, 按 `rest_id` 去重。

**防御性设计 (仍然要)**: `tweets.raw_payload jsonb` 存原始 `<TWEET>`; 解析层只取上表字段; 上游改结构时能从 `raw_payload` 重解析, 不丢数据。

⚠️ **作者路径按端点不同 (W1 实测发现, 解析器已做双路径兼容)**: `UserTweets`/`Search` 用 `core.user_results.result` (复数) + **新 schema** (`core.{name,screen_name}` + `avatar.image_url`); 而 **`TweetDetail` 用 `core.user_result.result` (单数) + 旧 schema** (`legacy.{name,screen_name,profile_image_url_https}`)。解析器两种都试 (见 [twtapi/parse.go](../../server/internal/infra/twtapi/parse.go) 的 `pickAuthor` + `toAccount` 新旧回退)。

### 3.4 增量采集策略 (没有 since_id, 自己做去重)

`UserTweets` **不支持 `since_id`**, 只有 `cursor` 翻页。所以增量逻辑自己做:

1. 每个账号存一个 **high-water mark** = 已采到的最新 `tweet_id` (Twitter id 单调递增, 可比大小; 字符串按数值比)。
2. 每轮 poll: 拉第 1 页, 自上而下遍历 entries:
   - `id > high_water` 且库里没有 → 插入新推文。
   - 命中已知/更旧的 id → **停止** (本页后面都是旧的)。
3. **置顶推文 / RT 会打乱「最新在前」的顺序** → 必须**按 `id` 去重 (唯一约束兜底), 不能只靠位置**。
4. 仅当**整第一页都是新推文** (说明停机太久积压) 才用 `cursor` 继续翻页**补抓**, 设安全上限 (如最多 5 页), 超出记一条 `log` 并放弃, 下轮再追。
5. 收尾: 更新 `high_water_tweet_id`、`last_polled_at`、(可选) 保存 `cursor`。

**UserTweets vs Search (实测后定论)**: 二者返回的单条推文结构一致 (§3.3), **`<TWEET>` 解析器共用一份**, 换源只换「取列表」那几行, 成本极低。

- **主力用 `UserTweets`**: 它是账号**真实时间线**, 最全最新。本产品的承诺是「订阅里的都读完了」, **完整性第一** —— 不能因为采集源漏推让用户的「已读完」是假的。P0 已把它的嵌套解析跑通 (有 fixture), 原来的「难解析」顾虑已消除。
- **`Search` (`q=from:{handle}&type=Latest`) 作为补充**: 它带 `_normalized` (好解析) + 直接用 handle (免解析 user_id) + 支持 `since:`/`until:` 日期算子 (`from:elonmusk since:2026-06-08`) 做天然增量。**最适合**: ① 订阅时按日期范围**初始回填**; ② 周期性**对账** (查 UserTweets 是否漏采)。
- ⚠️ **不要只图 Search 好解析就拿它当唯一源**: 搜索索引有滞后/可能漏最新推, 当唯一源会**静默丢推**, 砸掉产品承诺。`count` 也受上游限制。

### 3.5 配额、限流与成本

| 套餐 | 价格 | 月调用上限 | 速率 |
|---|---|---|---|
| 免费 | $0 | **300 次/月** | 100 次/小时 |
| 入门 | $12/月 (订阅价) | 50,000 次/月 | 100 次/秒 |
| 标准 | $36/月 | 150,000 次/月 | 100 次/秒 |
| 进阶 | $80/月 | 400,000 次/月 | 100 次/秒 |

计费 = **每次 API 调用计一次** (月配额制, 超额次月账单补)。**成本只跟「不同账号数 × 每天 poll 次数」走, 与用户数无关** (共享推文存储, 见 §4)。

成本估算 (设订阅的**去重后账号** = 50 个):

| poll 间隔 | 次/账号/天 | 月调用 ≈ | 够用套餐 |
|---|---|---|---|
| 每 60 分钟 | 24 | 50×24×30 = 36,000 | $12/月 (50k) ✅ |
| 每 30 分钟 | 48 | 72,000 | $36/月 (150k) |
| 每 15 分钟 | 96 | 144,000 | $36/月 (150k) |

- 免费套餐 (300/月) **只够开发自测** (1-2 个账号、每天几次)。
- **推荐**: 自适应间隔 —— 活跃账号 15-30 分钟, 安静账号退避到 1-3 小时 (见 §5.4)。内测 ≤5 人、几十个账号, **$12-$36/月** 足够。
- `UsernameToUserId` 每账号只调一次 (缓存), 可忽略。
- **DeepSeek 分类成本另算** (走自己的 LLM 预算): 每条新推文 1 次调用, 50 账号 × 约 5 推/天 ≈ 250 次/天 ≈ 7,500 次/月, 很便宜。

### 4.1 `twitter_accounts` — 被采集的账号 (全局共享, 去重单位)

```sql
CREATE TABLE twitter_accounts (
  id                   uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  rest_id              text UNIQUE NOT NULL,          -- twtapi user_id (Rest ID)
  handle               text NOT NULL,                 -- screen_name, 不含 @ (可能改名)
  handle_lower         text GENERATED ALWAYS AS (lower(handle)) STORED,
  display_name         text,
  avatar_url           text,
  bio                  text,
  high_water_tweet_id  text,                          -- 已采最新 tweet id (增量基准)
  last_polled_at       timestamptz,
  poll_interval_sec    int NOT NULL DEFAULT 1800,     -- 自适应间隔
  status               text NOT NULL DEFAULT 'active',-- active | suspended | not_found
  created_at           timestamptz NOT NULL DEFAULT now(),
  updated_at           timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_twitter_accounts_due ON twitter_accounts (last_polled_at)
  WHERE status = 'active';
```

### 4.2 `subscriptions` — 用户 ↔ 信号源 (per-user, **v1 起多态**)

为「订阅 Telegram 频道 / 新闻源」预留 (§0 已拍板): 订阅关系不绑死 twitter, 用 `(source_type, source_id)` 指向各类型自己的源表。代价是 `source_id` 没有数据库级 FK (多态做不了), 完整性靠应用层 + 各源表自身约束; 换来的是加类型零迁移。

```sql
CREATE TABLE subscriptions (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid NOT NULL REFERENCES users(id),
  source_type  text NOT NULL DEFAULT 'twitter'
               CHECK (source_type IN ('twitter','telegram','rss')),
  source_id    uuid NOT NULL,        -- → twitter_accounts.id (twitter) / 未来各类型源表.id
  created_at   timestamptz NOT NULL DEFAULT now(),
  active       boolean NOT NULL DEFAULT true,
  UNIQUE (user_id, source_type, source_id)
);
CREATE INDEX idx_subscriptions_user ON subscriptions (user_id) WHERE active;
```

每个类型 = 自己的源表 + 自己的采集器 + 自己的内容表 (`twitter_accounts` + `tweets` 是第一组; 未来 `telegram_channels` + `telegram_messages`、`news_feeds` + `news_articles` 同构复制), 见 §8.7。

### 4.3 `tweets` — 推文 (全局共享, 一条采一次/分类一次)

```sql
CREATE TABLE tweets (
  id                  text PRIMARY KEY,               -- tweet rest id (twtapi)
  twitter_account_id  uuid NOT NULL REFERENCES twitter_accounts(id),
  text                text NOT NULL,
  lang                text,
  tweet_created_at    timestamptz,                    -- 推文发布时间 (解析 twtapi created_at)
  is_retweet          boolean NOT NULL DEFAULT false,
  is_quote            boolean NOT NULL DEFAULT false,
  media               jsonb,                          -- [{type,url,width,height}]
  metrics             jsonb,                          -- {likes,retweets,replies,views}
  raw_payload         jsonb NOT NULL,                 -- 原始响应, 兜底重解析 (见 §3.3)
  -- AI 分类回写:
  tags                text[],
  summary             text,
  classify_status     text NOT NULL DEFAULT 'pending',-- pending | done | failed
  classified_at       timestamptz,
  captured_at         timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_tweets_account_time ON tweets (twitter_account_id, tweet_created_at DESC);
CREATE INDEX idx_tweets_classify_pending ON tweets (captured_at) WHERE classify_status = 'pending';
```

### 4.4 `tweet_reads` — 已读 (per-user, 非事件溯源)

已读是高频、低价值动作, 不进 events 表 (避免事件膨胀), 走普通 upsert:

```sql
CREATE TABLE tweet_reads (
  user_id   uuid NOT NULL REFERENCES users(id),
  tweet_id  text NOT NULL REFERENCES tweets(id),
  read_at   timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, tweet_id)
);
```

「某用户的某推文是否已读」= `tweet_reads` 是否存在该行。未读数 = 该用户订阅账号下的 `tweets` 减去 `tweet_reads` 的差集 (见 §6 计数查询)。

### 4.5 事件类型 (`server/internal/domain/event.go`)

仿 `EventSignalCaptured = "signal.captured"` 增加:

| 事件 | 何时发 | 谁消费 |
|---|---|---|
| `tweet.captured` | poller 插入新推文 | Mastra (分类) |
| `tweet.classified` | 分类写回后 (可选) | 暂无 / 未来推送 |
| `subscription.added` / `subscription.removed` | 订阅变更 | 审计 / 未来分析 |
| 转信号 → **复用 `signal.captured`** | 用户点「转为信号」 | 现有信号管线 (零改动) |

### 5.2 后台 poller worker

仿 [`server/internal/module/exit/checker.go`](../../server/internal/module/exit/checker.go) 的 ticker 循环 + [`infra/iii/outbox.go`](../../server/internal/infra/iii/outbox.go) 的 `SKIP LOCKED` 取任务, 在 `cmd/api/main.go` 用 `workerWG.Add(1); go poller.Run(ctx)` 起。

```go
func (p *Poller) Run(ctx context.Context) {
    tick := time.NewTicker(60 * time.Second) // 调度心跳, 不是每账号间隔
    for {
        select {
        case <-ctx.Done(): return
        case <-tick.C:    p.pollDueAccounts(ctx)
        }
    }
}

// pollDueAccounts: 取 status='active' 且 (now - last_polled_at) >= poll_interval_sec 的账号
// (SELECT ... FOR UPDATE SKIP LOCKED 限批, 避免多实例重复采), 逐个:
//   page := client.UserTweets(ctx, acct.RestID, "")
//   for _, t := range page.Tweets {  // 自上而下, 命中 high_water 即 break (见 §3.4)
//       repo.InsertTweetAndEvent(ctx, t) // tweets 行 + tweet.captured 事件 + outbox 行 (一事务)
//   }
//   repo.UpdateHighWater(ctx, acct.ID, newest, time.Now())
```

错误处理: `429` → 该账号本轮跳过 + 拉长 `poll_interval_sec` (退避); `402` → 记 ERROR + 全局暂停采集 (别空转烧配额); `404` → `status='not_found'`, 不再采。

`InsertTweetAndEvent` 与 [`signal/repository.go`](../../server/internal/module/signal/repository.go) 的 `Capture` 同构: 一个事务里写 `tweets` (ON CONFLICT (id) DO NOTHING 幂等) + `events('tweet.captured')` + `event_outbox` 行。outbox worker (已存在) 会把它 POST 到 Mastra (见 §5.3)。

### 5.3 事件 → Mastra 路由

[`server/internal/infra/iii/client.go`](../../server/internal/infra/iii/client.go) 的事件→路径映射表加一行:

```go
"tweet.captured": "/v1/events/tweet-captured",
```

outbox worker 自动把 `tweet.captured` 发到这个 Mastra HTTP shim, 余下交给 Mastra (§7)。

### 5.4 自适应 poll 间隔 (省钱)

`poll_interval_sec` 按账号实际产出动态调:

- 本轮采到 ≥1 条新推 → 缩短 (下限 900s = 15min)。
- 连续 K 轮空手 → 拉长 (上限 10800s = 3h)。

简单的「采到就提速、空轮就降速」即可, 不需要复杂预测。

### 5.5 转信号桥接 (零新管道)

`POST /v1/tweets/:id/promote` → service:

1. 读 `tweets` 行, 取 `text` + 作者 handle。
2. 调用现有 [`signal`](../../server/internal/module/signal) 模块的 `Capture`, `raw_text = text` (或加前缀 `via @handle:`), `client_event_id` 用 `tweet:{id}` 派生 (幂等, 同一推文不会转两次)。
3. (可选) 在 payload 里塞 `source: {type:"tweet", tweet_id, handle}` 留痕。
4. 返回新建的 `signal_id`, 客户端可跳转到信号详情。

信号被创建后, **整条现有管线 (推演 → 投决会 → 承诺) 自动接管**, 本功能不碰那部分。

### 5.6 配置 / 密钥

[`server/internal/config/config.go`](../../server/internal/config/config.go) 加:

```go
TwtAPIKey        string // os.Getenv("TWTAPI_API_KEY"), 缺失则 poller 不启动 (优雅降级)
PollEnabled      bool   // os.Getenv("SUBSCRIPTION_POLL_ENABLED"), 默认 true
```

`TWTAPI_API_KEY` 加进 [`.env.example`](../../.env.example) 和服务器 systemd env (见 [SERVER.md](../../SERVER.md))。Key 缺失时 `client.IsConfigured()==false` → poller 不起、REST 仍可读历史数据 (与 mastra client 降级一致)。

## § 6 · REST 契约 (移动端 ↔ Go)

全部挂 `publicV1` (`/v1`, Bearer 鉴权, 见 [auth model](../../server/internal/httpapi/auth/auth.go))。补进 [`docs/api/openapi.yaml`](../api/openapi.yaml)。

| Method + Path | 说明 | 关键字段 |
|---|---|---|
| `GET /v1/subscriptions` | 我的订阅列表 | 返回 `source_type` + 账号资料 + 每个的未读数 |
| `POST /v1/subscriptions` | 新增订阅 | body `{source_type:"twitter", handle}` → 解析→建 account (若新)→建 subscription→触发首次回填; 非 twitter 类型返 400「暂不支持」(为 Telegram/RSS 预留) |
| `DELETE /v1/subscriptions/:id` | 取消订阅 | 软删 (`active=false`) |
| `GET /v1/tweets` | 推文流 (feed) | query: `filter=unread\|all`, `subscription_id?`, `cursor?`, `limit?`; 返回带 `read`/`tags`/`summary` |
| `GET /v1/tweets/:id` | 推文详情 | 含 `raw`/thread (按需调 TweetDetail) |
| `POST /v1/tweets/:id/read` | 标记已读 | upsert `tweet_reads` |
| `POST /v1/tweets/read-all` | 全部已读 | query `subscription_id?` (不传=全部订阅) |
| `GET /v1/tweets/unread-count` | 未读总数 | tab 红点用 |
| `POST /v1/tweets/:id/promote` | 转为信号 | 返回 `{signal_id}` (见 §5.5) |
| `POST /v1/internal/tweets/classify-result` | **(internal)** Mastra 写回分类 | body `{tweet_id, tags, summary}` → 写 `tweets`, `classify_status=done` |

feed 查询 (未读优先) 形如:

```sql
SELECT t.*, (r.tweet_id IS NOT NULL) AS read
FROM tweets t
JOIN subscriptions s ON s.source_type = 'twitter' AND s.source_id = t.twitter_account_id AND s.user_id = $1 AND s.active
LEFT JOIN tweet_reads r ON r.tweet_id = t.id AND r.user_id = $1
WHERE ($2::text IS NULL OR s.id = $2)        -- 可选按订阅过滤
  AND ($3 = 'all' OR r.tweet_id IS NULL)     -- filter=unread 时只看未读
ORDER BY t.tweet_created_at DESC
LIMIT $4;                                    -- cursor 用 tweet_created_at + id 复合游标
```

未读数:

```sql
SELECT COUNT(*) FROM tweets t
JOIN subscriptions s ON s.source_type = 'twitter' AND s.source_id = t.twitter_account_id AND s.user_id = $1 AND s.active
LEFT JOIN tweet_reads r ON r.tweet_id = t.id AND r.user_id = $1
WHERE r.tweet_id IS NULL;
```

## § 8 · 移动端 UX 规格 (RN / Expo)

技术栈见 [06 RN 应用架构](06_ReactNative应用架构_大纲.md): expo-router、`ky` + TanStack Query、zustand、自定义排版 (Display/Serif/Mono) + `@expo/ui` 原生基座。设计 token 见 [`mobile/src/core/theme`](../../mobile/src/core/theme)。

视觉气质 = 全 app 的报刊感: 订阅 tab 是当年「报纸」占位的兑现 —— **你订的账号是通讯社, AI 是编辑部, 每天给你排一份只属于你的报纸**。

### 8.0 概念模型 · 订阅 = 多类型信号源的统一阅读面

三层概念, UI / 数据 / API 都按这个分层 (这是「第一版只有推特, 但不能长成只装得下推特」的结构保证):

| 层 | v1 | 未来 |
|---|---|---|
| **信号源类型** `source_type` | `twitter` (唯一) | rss / 公众号 / 财报日历 / 链上异动 … |
| **订阅项** subscription | 一个 X 账号 | 一个 feed / 一个号 / 一只标的 … |
| **内容项** item | 一条推文 | 一篇文章 / 一条公告 / 一次异动 … |

**统一契约**: 无论哪种类型, 每个内容项都享有同一套**四件套 —— AI 标签+总结 · 已读 · 转为信号 · 详情页**。四件套不随类型变, 阅读面才能跨类型统一。

**类型选择器渲染策略 (决策, 2026-06-10 修订)**: 阅读面和管理面分开处理 ——

- **阅读面 (feed) v1 不渲染类型行**: 只有一种类型时筛选器是死 UI。刊头下方预留插槽, 第二种类型上线时 chips 行 (`[全部] [X] [Telegram] [RSS]`) 直接出现, 零重构。
- **管理面 v1 即类型优先** (已拍板, 见 §0): 「添加订阅」先见类型选择 —— `X 账号` 可用, `Telegram 频道`、`新闻源 (RSS)` 以「规划中」灰态在场。灰态卡不是死 UI, 是**路线图的可视化**: 让用户知道这个面会长成什么样, 也倒逼数据模型从 v1 就多态 (§4.2)。订阅列表按类型分组。

- **全部已读** — 文字按钮, 仅未读视图且未读 >0 时出现; 点击 = `useMarkAllRead` 乐观清零 + 轻 haptic

**列表**: 按日分组 (`Mono` 小标日期头 `6 月 10 日 · 周三`), 组内时间倒序; FlatList + RefreshControl + cursor 加载更多; 底部留 `insets.bottom + 64` 给悬浮 tab bar。

**TweetRow 解剖** (报纸条目, 不是社交卡片 — 无头像大图、无互动计数):

```
▌ @handle · 3 小时前                    ◆     ← category 色标 | Mono 作者行 (RT 加前缀) | 未读标
  AI 总结当标题 (Serif semibold)               ← 长推: 编辑替你拟的题
  原文摘录最多两行, muted……   [图] [视频]      ← 媒体只给字形标记, 行内不放缩略图
  〔公司〕〔AI 芯片〕                            ← 自由标签 pills (hairline 描边, 仿 ArchiveView)
```

- **智能密度规则 (决策)**: 原文 ≤100 字 → 原文直接当标题, 不显总结 (短推的总结是同义反复); >100 字或带线程 → **AI 总结当标题**, 原文降为 muted 摘录; 总结未就绪/失败 → 原文当标题 + 行尾淡字「AI 正在读…」, 不阻塞阅读。
- **已读态**: 整行 ink→muted, ◆ 消失。
- **快捷手势**: 行左滑 →「已读」(不打开就归档的快路径); 长按 → 原生菜单 (已读 / 转为信号 / 在 X 打开)。

**空态 / 读完态** (编辑部口吻, 不卖惨不催促, 同旧报纸占位的腔调):

- 无订阅:「这份报纸还没有通讯社。」+ CTA「添加第一个账号」
- 未读清零: 居中 `Mono` stamp `✓ 都读完了` + `Serif italic`「今天订阅里的事, 你都过目了。」+ 轻 haptic —— **这是整个 tab 的情绪回报, 值得花心思做**。

### 8.3 管理订阅 (/subscriptions/manage) · 类型优先

页面从上到下: **类型选择区 → 当前类型的添加流 → 按类型分组的订阅列表**。

**类型选择区** (三张小卡横排; 每个类型自带自己的添加流):

| 类型 | v1 状态 | 添加流 | 采集端 (规划阶段的候选) |
|---|---|---|---|
| **X 账号** | ✅ 可用 | `@handle` → 解析预览 → 确认 | twtapi (本文档主体) |
| **Telegram 频道** | 「规划中」灰态 | `t.me/链接` 或 `@频道名` → 预览 | 公开频道: t.me/s/ 网页源 或 MTProto/Bot API, 立项时再选 |
| **新闻源 (RSS)** | 「规划中」灰态 | feed URL → 探测 title/icon → 预览 | 标准 RSS/Atom 拉取, 三者中最容易 |

灰态卡可见不可用 (tap 给一句「在路上了」)。它不是死 UI, 是路线图可视化, 同时倒逼多态数据模型 (§4.2)。

**添加流程 (X 账号, 一张 sheet 内 3 步)**:

1. 输入 `@handle` — `NativeField`, 自动剥 `@`/空格, 大小写不敏感
2. **解析预览卡** — 调 `UserResultByScreenName` 显示头像+名称+@handle+简介, 让用户确认"是这个人" (防错订重名号)
3. 确认 →「已订阅。正在回填最近的推文, 新内容会出现在你的报纸里。」

失败态: 404 / 受保护账号 → 预览卡位给一句话说明, 不弹窗。

**订阅列表** (按类型分组, v1 只有「X 账号」一组): 头像 · 名称 + @handle · 未读数 · 最近一条时间 · 失效徽标 (`not_found`/`suspended`)。左滑取消 (即时生效, 无二次确认 — 可随时订回, 不算破坏性动作)。

**软上限**: 上限按**类型**由 server 下发 (X 账号建议 30, 对应 §3.5 成本账; 未来 RSS 几乎零成本可放宽), 触顶文案:「先读完手头的, 再添新的。」—— 产品哲学顺手当成本闸门。

### 8.4 详情页 (/tweet/[id])

仿 [`mobile/app/signal/[id].tsx`](../../mobile/app/signal/[id].tsx) 的报刊版式:

- header: back + `Mono` stamp (`订阅 · @handle`)
- 作者块: 头像 + 名称 + @handle + 完整时间
- **全文**: `Serif` 16-17、行高放宽 — 当文章排, 不当推文排
- 媒体: 照片 inline (单图全宽 / 多图两列); **视频 v1 只放封面 + 播放角标, tap 跳 X** (省播放器复杂度)
- 引用推: 细描边嵌套卡片 (作者 + 摘录)
- 线程:「展开整条线程」按钮 → `GET /v1/tweets/:id` 懒调 TweetDetail
- AI 块: `DoubleRule` + `SectionHeader「AI 编辑」` + 总结 + 标签 pills + category
- **进入即已读** (`useMarkRead` 乐观更新, 无需手点)
- 底部动作区: **转为信号** (primary) · 在 X 打开 (secondary)

### 8.5 转为信号 (订阅 → 信箱的桥)

点「转为信号」→ confirmation sheet:

1. 推文引用卡 (作者 + 原文)
2. **可选一行输入**: 「用你的话说一句 (可选)」—— 信号的灵魂是*你的*「咦」, 不是别人的推。填了 → `raw_text` = 你的话 + 引用出处; 不填 → 原文直通 (前缀 `via @handle:`)
3. 确认 → `POST /v1/tweets/:id/promote` → sheet 内出现「去看信号 →」轻链接, 不强制跳转
4. **幂等**: 已转过的推文, 按钮态变「已转为信号 · 查看」

转出的信号与手录信号同进信箱、同走推演/投决会; 信号详情显示出处 `via @handle` (payload.source 已留, 见 §5.5)。

## § 9 · 关键决策与权衡 (ADR-style)

| 决策 | 选择 | 理由 / 否决项 |
|---|---|---|
| 采集 endpoint | `UserTweets` 主力, `Search(from:)` 补充 | 实测: 二者单条结构一致, **解析器共用一份**。UserTweets 完整性第一 (产品承诺); Search 有 `_normalized`+`since:`, 用于回填/对账, 但单用会静默丢推 (§3.4) |
| 增量方式 | high-water + 按 id 去重 + cursor 补抓 | 无 `since_id`; 置顶/RT 会乱序, 只能按 id 去重 |
| 推文存储粒度 | **全局共享** (per-account), 订阅/已读 per-user | 多人订同一账号只采一份、分类一次 → 省 twtapi + DeepSeek; 成本与用户数解耦 |
| 分类时机 | **异步** (iii 队列), 非同步 | 不阻塞采集; 与信号 `signal.captured→inference` 同构 ([ADR 0004](../adr/0004-iii-over-nats.md)) |
| 已读模型 | 普通表 upsert, **不进 events** | 高频低价值, 进 events 会污染事件流 |
| 转信号 | **复用 `signal.Capture`**, 不建新管道 | 零重复; 推文 → 信号 → 推演/投决会 自动接管 |
| 标签 | 固定 `category` 大类 + 自由 `tags` | 大类可聚合/可筛 (投资场景需要), 细标签保留灵活 |
| poller 位置 | **Go 后台 goroutine** | 与现有 outbox/exit worker 同形, 不引入新运行时 |
| 行内信息密度 | 长推 AI 总结当标题, 短推 (≤100 字) 原文直出 | 短推总结是同义反复; 长推的价值恰在「编辑拟题」(§8.2) |
| 类型选择器 | 阅读面 v1 不渲染 (留插槽); 管理面 v1 类型优先 (Telegram/RSS 规划中灰态) | 阅读面单选项筛选是死 UI; 管理面灰态卡是路线图可视化, 用户点名要预留 (§8.0/§8.3) |
| 订阅关系表 | v1 起多态 `(source_type, source_id)` | 加 Telegram/RSS 零迁移; 代价: `source_id` 无数据库 FK, 完整性靠应用层 (§4.2) |
| tab 未读提示 | 纯红点, 不带数字 | 克制: 数字制造焦虑; 具体数额放刊头 stamp (§8.1) |

## § 11 · 风险与未决问题

1. **🟢 响应结构 (P0 已消险)**: 原是头号风险 (无文档化), 现已实测固化字段映射 (§3.3) + 落地 fixture。残留: 上游 x.com 改 schema 的可能 → 靠 `raw_payload jsonb` 兜底重解析 + 解析器对作者新旧 schema 做兼容。
2. **🟡 单一第三方依赖**: twtapi 挂了 / 改结构 / 涨价 → 采集断。缓解: client 加超时+重试+告警; 解析层与 `Search` 备选共用, 可快速切换; `402/429` 有明确降级 (§5.1)。
3. **🟡 配额成本**: poll 越密越贵 (§3.5)。缓解: 自适应间隔 + 共享存储 (成本与用户数无关) + 配额告警。
4. **🟡 推文增删改**: 删推 / 编辑推 / 仅置顶变化。MVP: 不主动同步删除 (库里留存); 编辑推可能产生新 id。后续再议是否回收。
5. **🟢 RT / 引用推**: 决定是否采 RT (默认采, 标 `is_retweet`)、引用原推是否展开。P0 看清结构后定。
6. **🟢 分类质量**: DeepSeek 偶发标签飘。抽取式任务影响小, schema 强约束 + 2 次重试兜底; 失败不阻塞阅读。
7. **🟢 隐私/合规**: 存第三方公开推文用于个人阅读, 内测范围 (见 [TESTFLIGHT_PLAN](../TESTFLIGHT_PLAN.md) § 0 ≤5 人) 风险低; 公开化前再评估 ToS。

## 章节结构

### § 1. 这个产品最该测什么

呼应产品哲学,最该测的不是"功能能跑",而是:

1. **沉默逻辑不被破坏** — 四道门未过的场景,绝对不能有任何用户感知的输出
2. **判据稳定性** — 同一份历史数据,跑同版本规则,结果完全一致
3. **事件不可变性** — events 表写入后任何代码路径不能更改
4. **LLM 输出符合人设** — Editor Agent 不能说"会涨回来的"这种话

测试优先级直接服务于产品的"良心"。

### § 2. 测试金字塔(本项目变体)

```
        E2E (少量,关键流程)
       ────────────────────
      Integration (适量,跨服务)
     ──────────────────────────
    Unit (大量,规则引擎/领域模型)
   ────────────────────────────────
  LLM Snapshot (中量,Agent 输出回归)
 ──────────────────────────────────────
```

最底层多了一层 "LLM Snapshot" — 这是 LLM 产品特有的。

### § 3. 规则引擎单元测试

四道门规则的测试范式:

- 每条门规则配 **20+ 历史 fixture**,从真实 events 表导出脱敏数据
- 每个 fixture 标注 expected result(pass / fail + 原因)
- 规则修改前后跑全量,**任何 result 变化必须解释**

```go
func TestGate1_SignalThickness(t *testing.T) {
    fixtures := loadFixtures("gate1/")
    for _, fx := range fixtures {
        result := gate1.Evaluate(fx.Input)
        assert.Equal(t, fx.Expected, result, fx.Description)
    }
}
```

Fixture 文件组织:

```
/testdata/gates/
  gate1/
    01_three_independent_signals_pass.yaml
    02_three_same_source_fail.yaml
    03_two_signals_pending.yaml
    ...
```

### § 4. 领域模型单元测试

- Signal 状态机:`pending → inferred → in_cluster → archived`
- Commitment 状态机:`draft → signed → active → expired`
- Exit Condition 判断逻辑

每个状态机配合状态图测试覆盖率(每条边至少一条用例)。

### § 5. LLM 输出测试 (Snapshot + Property)

**Snapshot Test**:

- 固定输入 → 第一次运行存为 baseline
- 后续运行,输出与 baseline 做语义相似度比较(用 embedding 距离)
- 阈值 > 0.85 通过,< 0.85 人工 review

**Property Test**(更重要):

每个 Agent 列硬约束,测试这些约束是否被违反:

Socratic Interviewer:
- ✓ 输出不超过 2 句话
- ✓ 不包含 "建议"、"参考"、"答案" 这些词
- ✓ 一次只包含一个问号

Editor Agent:
- ✓ 必须包含承诺书里的至少一句原文
- ✓ 不包含 "会涨回来"、"别担心"、"放心" 这些词
- ✓ 不包含股价预测语言

```typescript
test('Editor never predicts price', async () => {
  const responses = await runEditor(20FixtureScenarios);
  for (const r of responses) {
    expect(r).not.toMatch(/会涨|会跌|大概率|很可能/);
  }
});
```

**Manual Eval**(定期跑):

- 每月抽 50 条 Agent 输出,人工评分
- 评分卡:角色一致性 / 是否违反硬约束 / 文案质量
- 评分下降是 prompt 漂移的早期信号

### § 6. 集成测试

测试跨服务的协作:

- POST /v1/signals → events 表写入 + NATS 消息发出
- NATS 消费 → Mastra Workflow 执行 → 推演结果回写
- 四门评估通过 → 承诺书草稿生成 → SSE 流式返回

每个集成测试用 docker-compose 起完整依赖,跑完销毁。

### § 7. E2E 测试

🟢 Phase 1 只覆盖一条 happy path:

**"用户录入信号 → 后台推演完成 → 进入观察池(沉默)"**

E2E 工具:Patrol(Flutter 集成测试框架),Newman(API 测试)。

后续 Phase 加:

- "信号簇升级 → 四门全过 → 用户签字 → 持仓状态"
- "持仓 30 天 → 用户多次打开 → E4 卡片触发 → 用户关闭卡片"
- "持仓到期 → 复盘对话 → 时间轴生成"

### § 8. 性能测试

Phase 1 不做正经的压测(单用户)。但要测:

- 单次信号录入 端到端延迟 < 300ms
- 五轮追问 首 token 延迟 < 1.5s
- 承诺书生成 完整文案 < 5s

测试工具:`vegeta` (HTTP),手动测 SSE。

### § 9. 测试数据管理

- 测试 fixture 与生产数据分离,用合成数据
- 真实事件流通过 `/v1/me/export` 导出 + 脱敏脚本,作为 baseline
- 不允许测试数据混入开发数据库

### § 10. CI 中的测试

GitHub Actions 跑:

- ✓ Go unit + integration (每次 PR)
- ✓ Flutter unit + widget (每次 PR)
- ✓ Mastra property test (每次 PR)
- ✓ Snapshot test (每天定时,LLM 调用花钱)
- ✗ Full E2E (手动触发,主分支合并前)

### § 11. Phase 1 覆盖率目标

不追求高覆盖率,但要求关键路径覆盖:

- Gate 1 规则:100% fixture 覆盖
- 信号状态机:100% 状态转移覆盖
- API 路径:Phase 1 路径全部有集成测试
- LLM Agent:每个 Agent 至少 5 个 property test
- E2E:至少 1 条 happy path

---

## 关键决策预告

1. **Fixture 驱动规则测试** — 规则不能脱离历史样本演化
2. **LLM 测试以 Property 为主,Snapshot 为辅** — 防漂移
3. **集成测试用真实容器,不 mock 依赖** — 配置错误最常见
4. **覆盖率不是目标,关键路径是** — 测得对比测得多重要

## 章节结构

### § 1. 产品哲学如何决定架构

不直接讲技术,先回到产品的几条反直觉原则,解释为什么它们排除了一些常见的架构选择:

- "沉默优于发声" → 排除了"AI 输出直接推给用户"的架构
- "频率匹配 alpha 频率" → 排除了高 QPS 假设
- "让用户看见自己" → 强制要求事件溯源
- "判据是产品良心" → 排除了"LLM 全权决定"的架构

### § 2. 五层架构 vs 技术分层

把产品的"五层架构"(L1 信号捕捉 → L5 复盘训练)和技术分层(客户端/网关/服务/LLM/存储)叠加起来,画一张二维矩阵,说明每个产品层在每个技术层里的具体存在形式。

### § 3. 三种数据流

整个系统只有三种数据流形态,所有 API 都是其中之一:

- **录入流** (同步,30 秒)
- **推演流** (异步,几分钟)
- **对话流** (同步流式,SSE)

### § 4. Go 与 Mastra 的契约边界

这一节是整份文档的核心。明确写出:

- Mastra 永远不直连数据库,只通过 Go 的 `/v1/internal/*` 接口
- Mastra Memory 只存会话上下文,不存事实
- Go 调 Mastra 用 HTTP,不嵌入 Mastra
- 双向调用的鉴权方式(用户 JWT vs service token)

### § 5. 状态分布

哪些状态在哪儿?

- 不可变事实 → PostgreSQL events 表
- 当前视图(派生) → PostgreSQL 物化视图
- 行为指纹 → Redis
- 对话上下文 → Mastra Memory
- 本地缓存 → Isar (Flutter)

### § 6. Phase 演进路径

### § 7. 几条留待后续讨论的问题 ⚠️

- 多用户场景下的隔离策略 (单机 vs 多租户)
- 跨设备同步 (Phase 3+)
- 数据导出与"产品消失"哲学的兼容(用户随时能带走自己的数据)

## 关键决策预告

这份文档会明确以下决策,后续文档遵循:

1. **事件溯源 + CQRS 模式**(Command Query Responsibility Segregation)
2. **Go 做规则引擎,Mastra 做语言转换**
3. **NATS JetStream 作为唯一的异步事件总线**
4. **PostgreSQL 单库多 schema,不引入额外数据库**

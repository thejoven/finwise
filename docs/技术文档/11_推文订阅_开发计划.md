## § 0 · 架构修订 (相对规格文档 §2 的一处简化)

规格 §2 原设计: `tweet.captured 事件 → event_outbox → iii 队列 → Mastra worker → POST /v1/internal 写回`。
**执行时修订为**: `poller 落库 (classify_status=pending) → Go 分类派发器 (dispatcher) → 同步 POST Mastra /tweet-classify → Go 直接 UPDATE tweets`。

理由 (与既有架构先例一致):

1. **events 表只装领域事件** —— [distillation 模块](../../server/internal/module/distillation/repository.go) 的明文先例: "派生分析投影不写 events 表"。推文是**全局系统数据** (非 per-user 领域事件), 硬塞进 per-user 的 events/outbox 管道要伪造 user_id, 弄脏事件溯源。
2. **at-least-once 语义不丢**: `classify_status=pending` + attempts 计数本身就是持久化队列 — 派发失败下轮重试, 与 outbox worker 的 drain 形态同构 (`SELECT … FOR UPDATE SKIP LOCKED`)。
3. **改动面更小**: 不动 iii 配置、不加 worker.ts 队列处理器/HTTP shim、不加 `/v1/internal/tweets/*` 端点。Mastra 只加一个无状态同步端点 (与 `/consensus-check` 等六个现成端点同形, 见 [http.ts](../../mastra/src/server/http.ts))。

订阅增删同理**不写 events** (表本身即记录, UNIQUE 约束即幂等)。`转为信号` 走 signal.Capture, 它内部自带 event+outbox, 不需要本模块做任何事件工作。

```
poller (60s tick) ──→ twtapi UserTweets ──→ tweets 表 (pending) ──┐
                                                                  │ dispatcher (5s tick, SKIP LOCKED)
mobile ←── REST /v1/subscriptions /v1/tweets ←── Go ←──┐          ▼
                                                       │   POST {MASTRA}/tweet-classify
                  signal 管线 (现有) ←── promote ──────┘          │
                                                                  ▼
                                              UPDATE tweets SET tags, summary, … done
```

## § 3 · W2 · subscription 模块

**文件**: `server/internal/module/subscription/{handler,service,repository,poller}.go`; 改 `config.go` `main.go`

**repository** (pgx 原生 SQL, 风格照 signal/distillation):

- `ResolveOrCreateAccount` (rest_id ON CONFLICT 返已有), `Subscribe`/`Unsubscribe` (UNIQUE 幂等, 多态 source_type)
- `ListSubscriptions` (join twitter_accounts + 未读数子查询)
- `InsertTweets` (batch, `ON CONFLICT (id) DO NOTHING`, 返新增数), `UpdateHighWater`
- `DueAccounts` (`FOR UPDATE SKIP LOCKED`, `now()-last_polled_at >= poll_interval_sec`), `AdjustInterval` (有新推→900s 下限, 连续空→×2 上限 10800s)
- `FeedPage` (规格 §6 SQL, 复合游标 `(tweet_created_at,id)`), `GetTweet`, `MarkRead`/`MarkAllRead` (upsert), `UnreadCount`
- `PendingForClassify` (SKIP LOCKED, attempts<3), `RecordClassifyResult` / `RecordClassifyFailure`

**service**: Subscribe 流程 = 输入校验 (剥@/小写) → twtapi 解析 (`UsernameToUserId`+资料) → repo 落库 → **立即触发该账号一次回填 poll** (goroutine); `source_type != "twitter"` → `ErrUnsupportedType` (400 文案「暂不支持」); Promote = 读 tweet → 组 `raw_text` (可选用户一句话 + `via @handle: 原文`) → 调注入的 `CaptureSignalFn` (main.go 闭包到 signalSvc.Capture), `client_event_id = uuid.NewSHA1(NameSpaceOID, "tweet-promote:"+userID+":"+tweetID)` 幂等; 软上限 30 (`ErrLimitReached`)。

**poller.go** (一个 Run 双 ticker, 照 [exit/checker.go](../../server/internal/module/exit/checker.go) 形态):

- 采集 tick 60s: `DueAccounts(批 10)` → 每账号 `UserTweets` 第一页 → 自上而下命中 high-water 即停 (id 数值比较, 置顶/RT 乱序靠 ON CONFLICT 兜底) → 整页全新才翻页补抓 (≤5 页) → 写库+更新水位+自适应间隔。429 → 跳过+拉长间隔; **402 → 全局暂停采集 60min + ERROR log**; 404 → `status='not_found'`。
- 分类 tick 5s: `PendingForClassify(批 4)` → 并发调 `mastraClient.ClassifyTweet` (新方法, 走现成 `post()` helper, 20s) → 成功 `RecordClassifyResult`, 失败 attempts+1 (≥3 → `failed`)。Mastra 未配置 → 跳过 (推文照常可读, 降级一致性)。

**handler** (`Register(publicV1)`, 路由表 = 规格 §6 减去 internal 行):
`GET/POST /subscriptions` · `DELETE /subscriptions/:id` · `GET /tweets` (filter/subscription_id/cursor/limit) · `GET /tweets/unread-count` · `GET /tweets/:id` · `POST /tweets/:id/read` · `POST /tweets/read-all` · `POST /tweets/:id/promote`

**config**: `TwtAPIKey` (`TWTAPI_API_KEY`, 可缺 — 缺则 poller 不启动仅 warn, REST 照常), `SubscriptionPollEnabled` (`SUBSCRIPTION_POLL_ENABLED` 默认 true)。**main.go**: repo/service/handler 接线 + `CaptureSignalFn` 闭包 + poller `workerWG.Add(1)`。

**验收**: `go build ./...`; `go vet ./...`; 新增单测 (cursor 编解码/高水位停止逻辑/promote 幂等键) 全绿。

## § 4 · W3 · Mastra 分类端点

**文件**: `mastra/src/agents/tweet-classifier.ts` (新) + `mastra/src/server/http.ts` (加路由)

- Schema: `{ tags: string[1..4], summary: string(8..120), category: enum[宏观,公司,行情,政策,技术,观点,其它], relevance: number(0..1) }`
- Agent instructions: 金融编辑视角; 短推 (≤100 字) summary 允许贴近原文; 不预测涨跌不荐股; 输出纯 JSON。runner 2 次重试, `temperature 0.2, maxTokens 500` (抽取式任务, 方差影响小 — 区别于 consensus 打分的噪声问题)。
- 路由 `POST /tweet-classify` body `{tweet_text, author_handle?, lang?}` → 200 结果 / 400 缺参 / 502 LLM 失败 (Go 侧计 attempt)。
- Go 侧: `infra/mastra/client.go` 加 `ClassifyTweet` (复用 `post()`, metricLabel "tweet_classify")。

**验收**: `npm run typecheck` 过; 部署后 curl 实测返回合法 JSON (W5)。

## § 7 · 风险执行注记

- **go.mod `go 1.24` vs 本机 go1.22**: 依赖 `GOTOOLCHAIN=auto` 自动拉 1.24; 失败则构建全部放 205 做。
- **twtapi 凭据**: API key 只存在 205 systemd env 与本机 shell, 任何文件改动提交前 `grep` 校验不含 key。
- **配额**: 测试期只订 1-2 个账号; poller 默认 30min 间隔, 回填一次 ≤1 页。402 全局暂停逻辑必须先于真实订阅上线。
- **凌乱并发**: poller/dispatcher 均 SKIP LOCKED, 多实例安全 (虽然 205 只跑单实例)。

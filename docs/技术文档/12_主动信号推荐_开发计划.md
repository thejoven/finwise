## § 2 · W1 · 迁移 + config

**迁移 `026_recommendations.up/down.sql`** ✅ 已落地。号定为 **026**:023=user_language、024=user_alpha_profile、**025=asset_tracking(姊妹「标的追踪」已占)**,故顺延到 026。表用规格 §5 的 `recommendations` DDL,P1 仅用 `context_type='commitment'`、`target_ref=commitment_id`:

```sql
CREATE TABLE recommendations (
    id, user_id, source_type DEFAULT 'tweet', source_id (→tweets.id),
    score real, rationale text, context_type text, target_ref uuid,
    status text DEFAULT 'pending' CHECK (pending|surfaced|dismissed|promoted|expired),
    model text, created_at, surfaced_at, acted_at,
    UNIQUE (user_id, source_id, context_type, target_ref)   -- 同位幂等
);
-- 索引: (user_id, context_type, target_ref) WHERE status IN ('pending','surfaced')
```
- 同 P0:派生投影,**不写 events**。`dismissed`=负反馈(builder 下次降权该 tag/源),`promoted`=强正反馈。

**config (`config.go` 新增,P1 首次需要)**:`REC_ENABLED`(默认 true)、`REC_RELEVANCE_MIN`(候选阈值)、`REC_PER_COMMITMENT_QUOTA`(默认 2/周)、`REC_CANDIDATE_WINDOW_DAYS`(默认 14)、`REC_CRON_INTERVAL`(默认 24h)。

## § 3 · W2 · recommend 模块扩展 (Go 漏斗)

新增 `builder.go`(§3 漏斗,worker 化,形态同 subscription poller:cron tick + `SELECT … FOR UPDATE SKIP LOCKED`):

1. **候选**:对每个活跃承诺,取近 `WINDOW_DAYS` 天、`relevance ≥ MIN`、未读/未推过、且**与命题相关**的已分类 tweets。相关性粗判:tweet.tags/text 命中命题 `asset` 或与命题领域(`user_alpha_profile.tag_affinity`)有交集。(13 标的归一落地后,改用 `signal_assets`/`asset_aliases` 做精确 ticker 匹配——见 §7 依赖。)
2. **粗排** (Go,吃 `user_alpha_profile`):`tag/category 亲和 × 命题相关度 × 时新性` → Top-K (≤10~20)。
3. **精排** (调 W3 Mastra):传 `{画像摘要, 命题(asset/action/exit_conditions/reasons), K 候选}`,收 `[{source_id, keep, score, rationale}]`。
4. **配额 + 沉默闸**:每命题 ≤`QUOTA`;不过线 → 丢弃,当天沉默(正确行为,非 bug)。
5. **写 `recommendations`**(幂等 UNIQUE)。

**端点 (`handler.go` 扩展)**:
- `GET /v1/commitments/:id/related` — 该承诺的相关情报(user-scoped,owner 校验复用闭包)。
- `POST /v1/recommendations/:id/dismiss`、`POST /v1/recommendations/:id/seen` — 反馈(dismiss→负反馈)。
- builder 由 cron tick 驱动;亦可在 profile rebuild 后增量触发。

## § 4 · W3 · Mastra recommender 精排端点

`mastra/src/agents/recommender.ts` + `/recommend` 端点(无状态,同 consensus-check 等六端点同形,见 `mastra/src/server/http.ts`):
- 入参 `{profileSummary, thesis:{asset,action,exit_conditions,reasons}, candidates:[{id,text,tags,category,summary}]}`;出参 `[{source_id, keep, score, rationale}]`。
- 注入 [market-context](../../mastra/src/agents/market-context.ts);提示词强调:**对这个用户、这条命题**是否 alpha + **反证优先** + **默认丢弃**(宁缺毋滥,呼应红线)。
- 复用每用户 `resourceId` working memory 读画像 / 回写透镜偏好(可回填 `user_alpha_profile.lens_preference`,P0 留空的那列)。

## § 7 · 风险执行注记

- **红线优先于召回率**:任何"为了多推"的改动都要先过 §0 红线。沉默是合格行为。
- **反证检测靠提示词**:recommender 要能区分"印证"与"反证",反证优先级更高(复盘最缺)。这是 W3 提示词调优的重点,需真实命题试跑。
- **依赖 13(标的追踪)**:精确的"命题↔标的"关联依赖 13 的 `signal_assets`/`asset_aliases` 归一。**P1 可先用粗匹配(ticker 子串 + 标签交集)上线**,13 到 P2 后再换精确匹配提升召回质量(规格 §2 客观地面真值在 P4 才咬合)。
- **冷启动**:`user_alpha_profile.sample_size` 低于阈值的用户,画像未成熟 → builder 可跳过或仅退化为 relevance 排序,且不强推。诚实"还不懂你"好过乱推。
- **迁移号**:本表已用 **026**(023=user_language、024=user_alpha_profile、025=asset_tracking 均已占);后续 recommend 迁移取 ≥027,与标的追踪 session 对齐避免撞号(见 [[signal-rec-and-target-tracking-dev]])。

## 章节结构

### § 1. 项目结构

采用六边形架构变体(Hexagonal / Clean),目录大致如下:

```
/cmd
  /api          # HTTP 服务入口 (含 in-process OutboxWorker, 把 event_outbox 投递给 iii)
  /migrate      # 数据库迁移工具
/internal
  /domain       # 纯领域模型, 不依赖任何外部包
  /usecase      # 用例编排
  /handler      # HTTP 入站适配器
  /repository   # 出站适配器 (Postgres / Redis / iii / Mastra HTTP)
  /platform     # 横切关注点 (日志、配置、限流)
/pkg            # 可被外部引用的工具
/ent            # Ent 生成的代码
```

### § 2. 核心领域模块

每个模块独立成包(`/internal/module/<name>`,内含 handler / service / repository),有清晰的入口和出口:

- **signal** — 信号录入与状态机(`inference` 推演接入已并入,reinfer + 回写)
- **gate** — 四道门规则引擎 (🟢 Phase 1 部分实现)
- **commitment** — 承诺书生命周期
- **companion** — E4 行为指纹与陪伴卡片触发(原 `anxiety`)
- **exit** — 退出条件巡检(原 `exit_monitor`)
- **retrospect** — 复盘四问(原 `retrospective`)

后续 Phase 落地时新增的模块:

- **refinement** — 五轮追问会话状态机(M5)
- **research** — 信号研究/RAG 资料接入
- **project** — 项目分组与项目级 `guidance`(≤2000 字);信号 capture 可带 `project_id`,由 `ProjectOwnerCheck` 校验归属
- **attention** — 五轮追问后的注意力诊断接入,写回 `focus / depth / breadth / execution` + insight + blindspot(M11-bis)
- **account** — 多用户注册/登录/session

每个模块给出:职责、依赖、典型 API、Phase 范围。

### § 3. 规则引擎设计

这一节专门讲四道门。重点说明:

- 规则用 Go 代码写,不引入 Drools / DMN 这种重型引擎
- 每条规则有 `version` 字段,演化时不覆盖旧版本
- 规则单元测试:每条门规则配 20+ 历史 fixture (从 events 表导出)
- 规则的 `Result` 是结构化的,带 `pass / fail / pending` 三态和原因

代码示例只到接口层面:

```go
type Gate interface {
    ID() string
    Version() int
    Evaluate(ctx context.Context, input GateInput) GateResult
}
```

### § 4. 异步事件管道 (iii 编排引擎)

早期用 NATS JetStream 直连;现已切换到 **iii 编排引擎 + transactional outbox**(详见 `docs/architecture-iii.html`):

- **写事件**:业务事务里同时写 `events`(append-only 事实)和 `event_outbox`(待投递),保证"业务落库即必投递"
- **OutboxWorker**:api 进程内每 500ms `FOR UPDATE SKIP LOCKED` 拉一批,按 `subject` POST 到 iii 的 HTTP shim(`/v1/events/*`)
- **iii engine**:把请求 enqueue 到 4 个命名队列(`signal-inference` / `refinement-step` / `attention-analyze` / `commitment-draft`),at-least-once,retry=3,失败进 DLQ(file 持久化)
- **gate 评估**:走 outbox 的 PostPublish 回调内联执行(替代原 NATS gate consumer)
- iii engine 重启不影响 Mastra:SDK worker 经 WS 自动重连并重新 register

### § 5. Mastra 客户端

调 Mastra 不是简单的 HTTP client:

- 重试 + 超时(LLM 调用可能慢)
- Streaming 响应转 Go channel
- 工作流状态查询(Mastra Workflow Run ID 追踪)
- Mock 模式(本地开发不依赖真 Mastra)

### § 6. 配置与密钥

- 配置用 `koanf`,支持 YAML + ENV 覆盖
- 密钥(LLM API key、JWT secret)从环境变量读
- 本地开发用 `.env`,生产环境用 Docker secrets

### § 7. 日志与可观测性

- 结构化日志 (zap),每条 log 带 trace_id 和 user_id
- OpenTelemetry 全链路追踪(React Native → Go → Mastra → iii)
- 关键业务指标(信号入库速率、门评估通过率、沉默率)用 Prometheus

### § 8. Phase 1 最小落地清单

🟢 Phase 1 实际要实现的最小集:

- signal 模块完整
- inference 模块接入
- knowledge_graph 模块的基础查询
- gate 模块只实现门 1(信号厚度)
- 其他模块只建包结构,留 TODO

## 关键决策预告

1. **Ent ORM,不用 GORM** — JSONB 强类型支持是刚需
2. **不引入 Drools/CEL,规则用纯 Go** — 单元测试和演化追溯远比"动态加载规则"重要
3. **业务事务与消息发布分离** — `events` + `event_outbox` transactional outbox,由 in-process OutboxWorker 投递 iii
4. **领域模型不依赖框架** — Gin / Ent / iii client 都是适配器

## 章节结构

### § 1. 项目结构

采用六边形架构变体(Hexagonal / Clean),目录大致如下:

```
/cmd
  /api          # HTTP 服务入口
  /worker       # 后台 NATS 消费者入口
  /migrate      # 数据库迁移工具
/internal
  /domain       # 纯领域模型, 不依赖任何外部包
  /usecase      # 用例编排
  /handler      # HTTP / NATS 入站适配器
  /repository   # 出站适配器 (Postgres / Redis / Mastra HTTP)
  /platform     # 横切关注点 (日志、配置、限流)
/pkg            # 可被外部引用的工具
/ent            # Ent 生成的代码
```

### § 2. 八个核心领域模块

每个模块独立成包,有清晰的入口和出口:

- **signal** — 信号录入与状态机
- **inference** — 推演结果接入
- **gate** — 四道门规则引擎 (🟢 Phase 1 部分实现)
- **commitment** — 承诺书生命周期
- **anxiety** — E4 行为指纹与陪伴卡片触发
- **exit_monitor** — 退出条件巡检
- **retrospective** — 复盘四问
- **knowledge_graph** — 个人知识图谱聚合查询

每个模块给出:职责、依赖、典型 API、Phase 1 范围。

### § 3. 规则引擎设计

这一节专门讲四道门。重点说明:

- 规则用 Go 代码写,不引入 Drools / DMN 这种重型引擎
- 每条规则有 `version` 字段,演化时不覆盖旧版本
- 规则单元测试:每条门规则配 20+ 历史 fixture (从 events 表导出)
- 规则的 `Result` 是结构化的,带 `pass / fail / pending` 三态和原因

代码示例只到接口层面:

```go
type Gate interface {
    ID() string
    Version() int
    Evaluate(ctx context.Context, input GateInput) GateResult
}
```

### § 4. 事件总线封装

NATS 直接用太裸,需要一层薄封装:

- 发布 API: `bus.Publish(ctx, event)` — 事件自动加 correlation_id、user_id
- 订阅 API: `bus.Subscribe(subject, handler)` — handler 自动包入 panic recovery + retry
- Outbox 模式:Phase 1 用 in-process,Phase 2 升级为 events 表 outbox

### § 5. Mastra 客户端

调 Mastra 不是简单的 HTTP client:

- 重试 + 超时(LLM 调用可能慢)
- Streaming 响应转 Go channel
- 工作流状态查询(Mastra Workflow Run ID 追踪)
- Mock 模式(本地开发不依赖真 Mastra)

### § 6. 配置与密钥

- 配置用 `koanf`,支持 YAML + ENV 覆盖
- 密钥(LLM API key、JWT secret)从环境变量读
- 本地开发用 `.env`,生产环境用 Docker secrets

### § 7. 日志与可观测性

- 结构化日志 (zap),每条 log 带 trace_id 和 user_id
- OpenTelemetry 全链路追踪(Flutter → Go → Mastra)
- 关键业务指标(信号入库速率、门评估通过率、沉默率)用 Prometheus

### § 8. Phase 1 最小落地清单

🟢 Phase 1 实际要实现的最小集:

- signal 模块完整
- inference 模块接入
- knowledge_graph 模块的基础查询
- gate 模块只实现门 1(信号厚度)
- 其他模块只建包结构,留 TODO

---

## 关键决策预告

1. **Ent ORM,不用 GORM** — JSONB 强类型支持是刚需
2. **不引入 Drools/CEL,规则用纯 Go** — 单元测试和演化追溯远比"动态加载规则"重要
3. **业务事务与消息发布分离** — Phase 2 起用 outbox 模式
4. **领域模型不依赖框架** — Gin / Ent / NATS 都是适配器

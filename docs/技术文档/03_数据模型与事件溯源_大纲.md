## 章节结构

### § 1. 事件溯源是什么(简要)

如果你没用过 ES,这一节给出最小心智模型:

- 事件是不可变的事实("用户在 X 时刻做了 Y")
- 当前状态是事件流的折叠结果
- 时间旅行:任何历史时刻的状态都可以重建

特意说明 财富密码 **不用纯 ES**,而是 **事件流 + 物化视图** 的实用变体——读路径不重放事件,而是查物化视图;只在复盘和审计场景重放。

### § 2. events 表完整 schema

```sql
CREATE TABLE events (
    id              BIGSERIAL PRIMARY KEY,
    user_id         UUID NOT NULL,
    client_event_id UUID UNIQUE NOT NULL,
    type            TEXT NOT NULL,
    payload         JSONB NOT NULL,
    occurred_at     TIMESTAMPTZ NOT NULL,
    recorded_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    causation_id    BIGINT REFERENCES events(id),
    correlation_id  UUID,
    related_asset   TEXT,
    related_thesis  TEXT
);
```

每个字段的设计意图、索引策略、为什么是 BIGSERIAL 不是 UUID 主键。

### § 3. 事件类型完整目录

按五层产品分组,把所有事件类型列出来,**每种都给出 payload 的 typed struct**:

**L1 信号层**
- `signal.captured` — 快速录入
- `signal.refined` — 五轮追问完成
- `signal.tagged` — Mastra 推演打标完成

**L2 训练层**
- `training.question_answered` — 选择题作答
- `training.ability_map_updated` — 能力地图更新

**L3 评估层**
- `gate.evaluated` — 每位分析师评估
- `gate.cluster_promoted` — 信号簇升级到下一池

**L4 承诺层**
- `commitment.drafted` — 草稿生成
- `commitment.signed` — 签字
- `commitment.regret_moment` — 想反悔(从 app_opened 派生)
- `commitment.exit_condition_checked` — 退出条件巡检

**行为层**(横切)
- `app.opened` — 打开 APP
- `app.session_ended` — 会话结束

**L5 复盘层**
- `retrospective.started` — 进入复盘
- `retrospective.diagnosed` — 四问诊断完成
- `retrospective.focus_set` — 下次训练重点写入

每种 payload 给出 JSON Schema 和 Go struct。

### § 4. 物化视图设计

不是所有视图都用 PostgreSQL 的 MATERIALIZED VIEW(它刷新慢)。用三种方式:

- **常规表 + 触发器更新** — signals, commitments
- **常规表 + 事件消费者更新** — holdings, ability_map
- **PostgreSQL MATERIALIZED VIEW** — 离线报表场景(暂不需要)

每种物化视图给出表结构和派生逻辑。

### § 5. pgvector 集成

向量怎么存,放哪张表:

```sql
CREATE TABLE signal_embeddings (
    signal_id UUID PRIMARY KEY,
    embedding vector(1536),  -- OpenAI text-embedding-3-small
    embedded_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON signal_embeddings USING hnsw (embedding vector_cosine_ops);
```

什么时候 embed(信号 refined 之后)、谁来 embed(Go 调 embedding API 写入)、查询时怎么用(Mastra 通过 `/v1/internal/rag/search` 调)。

### § 6. 不可变性保护

防止 events 表被误改:

- 数据库层面:`REVOKE UPDATE, DELETE ON events`,只能 INSERT 和 SELECT
- 应用层面:Ent 生成的 Event 客户端只暴露 Create + Query,无 Update
- 数据修正:错误事件用"补偿事件"覆盖,不删原事件

### § 7. 迁移策略

Phase 1 用 `golang-migrate`,纯 SQL 文件。

迁移哲学:**只加列、只加表、只加索引**,不删不改。Phase 2 起 schema 进入冻结期,变化通过新事件类型实现。

### § 8. 数据导出(用户的权利)

呼应产品哲学 "alpha 来源是你自己"——用户应该能随时把自己的数据带走:

- 提供 `/v1/me/export` 接口,返回所有 events 的 JSONL
- 支持选择性导出(只信号 / 只承诺 / 完整)

---

## 关键决策预告

1. **BIGSERIAL 主键 + UUID client_event_id** — 主键查询高效,幂等用 UUID
2. **JSONB payload + 应用层强类型** — 灵活性 + 类型安全
3. **物化视图通过事件消费者更新,不用数据库触发器** — 触发器调试痛苦
4. **embeddings 与 events 表分离** — embedding 模型升级时只重新生成 embeddings
5. **REVOKE UPDATE/DELETE 是硬保护** — 事件溯源的不可变性必须由数据库层保证

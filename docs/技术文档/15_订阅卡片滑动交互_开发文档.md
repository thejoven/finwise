## § 0 · 一句话与它为什么重要

> 把订阅 feed 从"读报"变成"分拣"。每条推文是一张卡, 三个方向各对应一个决定:
> **左=已读放过, 右=转成信号收进信箱, 下=不感兴趣(并教系统少推同类)**。
>
> 产品名: **资讯工作台**(原"创新阅读模式")—— 每次标记(转信号 / 已读 / 不感兴趣)都是推荐流的训练信号, 你标记得越多, 推荐越懂你(推荐算法见 [12 主动信号推荐](12_主动信号推荐_开发文档.md); 本文档只管标记的**捕获**与负反馈减噪)。

为什么值得做:

- **降低处理成本**。现有列表要点进详情才能转信号, 一条要两三次交互。卡片台把"看一眼→做决定"压成一次滑动, 形成肌肉记忆, 适合每天清几十条订阅的高频场景。
- **负反馈闭环**。列表 feed 是被动的——你读或不读, 系统学不到你的口味。下滑"不感兴趣"第一次让用户能**主动塑形** firehose: 按内容标签累积厌恶, 后续 feed 自动减噪。这是订阅从"全量推送"走向"懂你"的第一步。
- **与既有立场一致**。promote 幂等管线、tweet_reads 已读模型、报刊式视觉都不动; 这一刀只在**交互层**与**一张反馈表**上加东西, 风险可控。

与产品口径的关系: 卡片仍是报刊式克制(衬线正文 + 克制留白), 不是花哨的社交流。"不感兴趣"的文案中性("减少此类"), 不挑衅。

## § 3 · 硬问题(诚实面对)

### 硬一 · 三向手势与点击/系统返回手势的冲突

卡片要同时支持: 水平拖(左右)、垂直下拖(下)、点击(进详情)。冲突点:
- 卡片台不需要竖向滚动(一次一张), 所以**纵向手势空出来给"下滑"**, 不与 ScrollView 抢。
- iOS 左缘是系统返回手势 → `Gesture.Pan().activeOffsetX([-12,12])` 设激活阈值, 避免误触; 卡片不贴左缘(留 14pt 边距)。
- 点击 vs 拖拽: 用 `Gesture.Exclusive(pan, tap)` + Pan 的 `activeOffset` 让小位移判为 tap、越过阈值才判为 drag。

**方案**: 单个 `Gesture.Pan`(同时跟 dx/dy) + 一个 `Gesture.Tap`, 用 `Exclusive` 组合; 释放时按 dx/dy/速度判方向(见 §4.1)。

### 硬二 · "不感兴趣"反对的是**内容标签**, 不是分类

⚠️ 关键澄清(用户明确要求): 下滑不是"我不看科技了", 而是"这**条**内容我不感兴趣, 据其 AI 标签判断, 以后少推**标签相近**的"。

- 触发: 下滑推文 T → 取 `T.tags`(AI 已生成的细标签, 如 `#美联储 #点阵图`)→ 每个标签 +1 厌恶权重。
- 不用 `category`(太粗: 一刀切掉整个"宏观"是误伤)。用 `tags`(细粒度, 可叠加, 多条命中同标签才"静音")。
- 阈值静音: 单个标签厌恶权重 ≥ `MUTE_THRESHOLD`(默认 2) → 标记 `muted` → 后续 feed 隐藏带该标签的推文。一次下滑不会立刻封一个标签(避免过敏), 累积才生效。
- **可恢复**: 提供"已静音标签"管理入口, 一键取消静音, 否则误伤无法回头(§10 开放项)。

### 硬三 · 卡片台 vs 列表 —— feed 形态取舍

卡片台(一次一张)聚焦、决断快, 但**丢失概览**(扫不到"今天有哪些大事")。列表相反。

**方案**: 订阅 tab 主形态改为**卡片台**(用户明确要的), 但保留两个出口补概览:
- 卡片右上"撤销"可往回翻最近几张;
- (P1.5 可选) 顶部"列表视图"切换, 退回旧 `TweetRow` 列表给想扫读的人。
默认卡片台。列表降级为可选。

### 硬四 · 撤销 —— 不做反向端点, 用"延迟提交 + 撤销窗口"

read 是幂等 upsert、promote 会真的建信号并触发推演——**撤销若靠反向端点, 要写 unread / 删信号, 复杂且危险**(删信号会牵连下游 inference/outbox)。

**方案(Gmail "撤销发送"式)**: 滑动只做**乐观 UI**(卡片飞走 + 顶部 toast 带"撤销"), 真正的服务端调用**入队延迟 ~2.6s** 再提交。窗口内点撤销 = 取消队列任务, 卡片弹回, **服务端从未被调用**。窗口过 = 静默提交。

收益:
- 三个动作(read/promote/not-interested)**统一**拿到撤销, 零反向端点。
- promote 不会产生"建了又要删"的信号。
- 离开页面 / 卡片清空时 flush 队列, 保证不丢动作。

代价: 动作有 ~2.6s 延迟落库(对这些非实时操作无感)。连续快速滑动时, 队列按 tweet_id 去重 + 顺序提交。

### 硬五 · 卡片"相关标的"情报层的数据来源

原型里卡片挂了 `$NVDA ★追踪` + "你在追踪英伟达·1 条相关信号"。但**推文表没有结构化标的**——`assets`/`signal_assets` 是**信号**推演后才有的([025_asset_tracking](../../server/migrations/025_asset_tracking.up.sql))。推文只有自由文本 `tags`。

**方案**: 分两步, 别在 P0/P1 阻塞。
- 现在(P0/P1): 卡片只展示已有的 `tags` + `category` + 来源, **不展示标的**。
- 以后(P2): mastra `tweet-classify` 步骤增补"标的抽取"(复用 signal 推演同款 ticker 归一化 + `asset_aliases`), 回写推文级 `tweet_assets`; 卡片再 JOIN `assets` + 用户追踪态渲染 ★。属增量, 不改前两期交互。

## § 4 · 交互 UX 与流程

### § 4.1 · 三向手势映射

| 手势 | 动作 | 端点 | 语义色 | 反馈 toast |
|---|---|---|---|---|
| **左滑** | 已读 | `POST /tweets/:id/read` (已有) | 中性墨色 | 「已读 · 归入已读」 + 撤销 |
| **右滑** | 转信号 | `POST /tweets/:id/promote` (已有, 空 note) | 绿 | 「已落入信箱 · 与手录信号同列」 + 去看信号 / 撤销 |
| **下滑** | 不感兴趣 | `POST /tweets/:id/not-interested` (**新**) | 红 | 「按内容减噪 · 少推 #tagA #tagB」 + 撤销 |
| 点击卡片 | 进详情 | 路由 `/tweet/[id]` (已有) | —— | —— |
| **上滑** | 稍后读 | `POST /tweets/:id/save` (**新**, = 标记已读 + 存稍后读) | 中性 | 「已存稍后读」 + 撤销 |

判定(释放时): `ax=|dx|`。
```
if (dy>0 && dy>ax && (dy>120 || vy>0.55))      → 'down'     // 不感兴趣
else if (dx>0 && ax>=dy && (dx>95 || vx>0.55)) → 'right'    // 信号
else if (dx<0 && ax>=dy && (dx<-95||vx<-0.55)) → 'left'     // 已读
else                                            → 回弹
```
即: 距离过阈值**或**甩动速度够快都触发(快甩不必拖满); 否则带背后卡片联动地弹回。

### § 4.2 · 卡片正文(明确: 不显示互动数, 显示来源)

从上到下:
1. **分类 pill** + 相对时间(右)。分类色按 `category` 取色(宏观/A股/科技/大宗/加密…)。
2. **作者**: 头像(首字占位)、display_name、`@handle`。
3. **AI 提炼**眉标(当有 `summary`): 小字"✦ AI 提炼"。
4. **标题**: 智能密度——`text` 长(>100 字)且有 `summary` → 用 summary 当标题、原文降为引用块;否则原文直接当标题。(沿用现有 [TweetRow](../../mobile/src/features/subscriptions/TweetRow.tsx) 逻辑)
5. **原文引用块** / **引用推嵌套块**(`is_quote` 时展示被引用原推)。
6. (P2) 相关标的 chips + 关联线。
7. **AI 标签** chips (`tags`)。
8. **来源页脚** (⚠️ 取代互动数): `来源 · X · 在 X 打开 ↗`。
   - ❌ **不显示** likes / retweets / views(数据库 `metrics` 里有, 但**不上屏**)。
   - 来源对多态数据源(X / 未来 Telegram·RSS, 见 [subscription-feature 备注])预留: 文案取 `subscription.source_type`。当前恒为 X。

### § 4.3 · 撤销 + 反馈 toast

- 每次滑动: 卡片飞出 + **屏顶居中简洁 toast**(深色胶囊: 动作结果 + 撤销, 已落地) 显示结果; 服务端动作入队, 2.6s 后提交。
- 撤销: 卡片从飞出方向弹回栈顶, 取消队列任务。
- 顶部"撤销"pill: 翻回上一张(同上)。
- 队列在以下时机 flush: toast 超时 / 切走 tab / 卡片清空 / 组件卸载。

### § 4.4 · 边界态

- **加载**: 报刊式"正在排版…"。
- **清空**: "✓ 都读完了 / 今天订阅里的事, 你都过目了。"(沿用 `empty.allRead` 文案)。
- **到底翻页**: 栈剩 ≤2 张时后台 `fetchNextPage`(无限滚动同 key)。
- **AI 未分类**: `classify_status==='pending'` → 标签位显"AI 正在读…", 标的/总结待回填(沿用现有轮询)。

## § 5 · 负反馈模型(落到代码)

### 触发链
```
下滑推文 T (tags=[a,b,c])
  └─ POST /v1/tweets/:id/not-interested
       ├─ 校验 T 在该 user 订阅范围 (GetTweet 复用)
       ├─ 事务:
       │    INSERT tweet_feedback (user, T, 'not_interested')  ON CONFLICT DO NOTHING  -- 隐藏这一条
       │    FOR tag IN T.tags:
       │      UPSERT user_tag_aversion (user, tag) weight += 1
       │    UPDATE user_tag_aversion SET muted=true WHERE user=$1 AND weight >= MUTE_THRESHOLD
       │    MarkRead(user, T)   -- 不感兴趣也算读过
       └─ 返回 { muted_tags: [...] }   -- 客户端可提示"已少推 #x"
```

### feed 过滤(P1 在 feed 查询加两条)
在 [subscription/repository.go](../../server/internal/module/subscription/repository.go) 的 feed 查询 WHERE 追加:
```sql
-- 隐藏明确说过不感兴趣的具体推文
AND NOT EXISTS (
  SELECT 1 FROM tweet_feedback f
  WHERE f.user_id = $1 AND f.tweet_id = t.id)
-- 隐藏带"已静音标签"的推文(标签级减噪)
AND NOT EXISTS (
  SELECT 1 FROM user_tag_aversion av
  WHERE av.user_id = $1 AND av.muted AND av.tag = ANY (t.tags))
```

### 阈值与口径
- `MUTE_THRESHOLD = 2`(常量, 可调): 同一标签被"不感兴趣"命中 2 次才静音, 防一次误伤封一片。
- 静音是**隐藏**(硬过滤), 不是降权排序——feed 仍按时间 cursor, 不重排(重排与分页 cursor 冲突, 留作 P2 推荐侧的事, 归 [12](12_主动信号推荐_开发文档.md))。
- **可恢复**: "已静音标签"管理页可取消 `muted`/清零 `weight`(P1.5)。

## § 6 · 数据模型 —— migration 031

新建 `server/migrations/031_tweet_feedback.up.sql` / `.down.sql`(接最新 030):

```sql
-- 031_tweet_feedback.up.sql
-- 订阅卡片"不感兴趣": 既隐藏当条, 也按内容标签累积厌恶, 减少后续同类推送.

CREATE TABLE IF NOT EXISTS tweet_feedback (
    user_id    uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    tweet_id   text NOT NULL REFERENCES tweets(id) ON DELETE CASCADE,
    kind       text NOT NULL CHECK (kind IN ('not_interested')),
    created_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, tweet_id)
);

CREATE TABLE IF NOT EXISTS user_tag_aversion (
    user_id    uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    tag        text NOT NULL,
    weight     int  NOT NULL DEFAULT 0,        -- 每次命中 +1
    muted      boolean NOT NULL DEFAULT false, -- weight ≥ 阈值 → 静音
    updated_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, tag)
);
CREATE INDEX IF NOT EXISTS idx_tag_aversion_muted
    ON user_tag_aversion (user_id) WHERE muted;
```
```sql
-- 031_tweet_feedback.down.sql
DROP TABLE IF EXISTS user_tag_aversion;
DROP TABLE IF EXISTS tweet_feedback;
```

字段沿用仓库惯例: `ON DELETE CASCADE`、`CHECK` 枚举、复合主键(见 [019](../../server/migrations/019_subscriptions.up.sql))。

## § 9 · 数据诚实、口径、失败模式

- **撤销延迟内崩溃/杀进程**: 队列未提交即丢。可接受(动作非关键, 下次仍在 feed)。组件卸载 flush 兜底。
- **promote 幂等**: 右滑同一条多次 → 同 SHA1 键 → 同一 signal(不会重复建)。卡片划走后通常不再出现, 重复概率低。
- **静音过敏**: 标签太泛(如 `#市场`)被静音会大面积误伤。缓解: 阈值≥2 + 管理页可解除 + (P2)考虑标签 IDF 权重, 泛标签不计入。
- **classify 未完成就下滑**: `tags` 为空 → 无标签可累积 → 只隐藏该条, 不影响标签厌恶。诚实记录: 此时负反馈"只对这一条生效"。
- **feed 过滤与 unread-count 一致性**: 过滤条件须同时进 feed 与 `unread-count` 查询, 否则角标与列表对不上。

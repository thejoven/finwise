§ IIIAPP 端 (Expo · /mobile)

M3 客户端外壳完成度很高, 字体/主题/组件/路由都按 spec. 但 M4 spec 要求的"持久化离线队列"几乎没做: pending 信号只在 zustand 内存, 重启就丢; 网络断开恢复也不会自动重传. 这是 Phase 1 验收 (W8 自己用一周) 真正过不了的硬伤.

严重度问题位置 / 影响
Critical 没装 expo-sqlite + drizzle-orm, 没有本地持久化
mobile/package.json (依赖未装) · mobile/src/features/capture/store.ts:1-13 (代码自承认)
M4 spec §4.1-4.2 明确要求. 当前 pending/failed 队列全部在 zustand 内存里, App 杀掉或重启就全没. 离线录入再联网"自动同步"完全做不到.
Critical 没装 @react-native-community/netinfo
mobile/package.json
M4 spec §4.4 要求, 用来监听网络变化触发自动 flush. 没装 → 用户必须手动下拉刷新才能触发同步. 这是 "离线录入再联网自动同步" 验收的核心.
Critical Sync Queue 只在冷启动跑一次
mobile/src/features/capture/PendingFlush.tsx:16-30 (代码注释自承认)
didFlushRef 保证只跑一次. 没监听 AppState (后台→前台), 没监听 NetInfo (断网→重连), 没监听 push notification (Phase 1 by design 不要). 用户实际体感: 录入失败后只能手动重试.
High 没有 max-attempts 上限, attempts 字段只增不停
mobile/src/features/capture/store.ts:42-57
M4 spec "重试 3 次, 然后标记 failed". 当前 markFailed 只 attempts+1, 没 cap, 没 backoff. 一个永远失败的请求会无限重试.
High capture 提交不"真正"先持久化再同步
mobile/app/capture.tsx (依赖内存 store)
UI 表现是"立即返回 inbox 看到记录" — 这是对的; 但底层只是 zustand upsert, 没落 SQLite. 用户在网络请求未完成时杀掉 App → 信号永远丢失. 这是离线优先的物理保证.
Medium Archive Tab 是 Phase 2 占位
mobile/app/(tabs)/archive.tsx:10-12 (代码注释明示)
Phase 1 by design 占位, 显示"承诺书归档 → Phase 2, 持仓陪伴 → Phase 3". 不算 bug, 但用户视觉上是"两个 tab 都好像能用".
Medium Masthead 菜单 / 搜索图标是 no-op
mobile/src/shared/components/Masthead.tsx:31 (代码注释明示)
≡ / ⌕ 是 Phase 2+ 抽屉/搜索入口. Phase 1 不传 onMenuPress 就 no-op. by design, 但 UI 上能看见图标会让用户尝试点.
Medium 主题没暗色模式
mobile/src/core/theme/index.ts:1-6 (代码注释明示)
Phase 1 only supports light; Phase 2 再加. 不算 bug, 但 iOS 用户开系统暗色时, App 还是亮色.
Medium 推演状态轮询频率硬编码 10s
mobile/src/features/capture/hooks.ts:35 (refetchInterval: 10_000)
M4 spec 注 "10 秒一次, 别太频繁". 但没有自适应 — 即使没有 pending 信号也每 10s 拉一次, 浪费电. 应该: 有 pending 才轮询, 全 done 后停.
Low capture 提交无 inline 反馈
mobile/app/capture.tsx (modal 直接 router.back())
符合 "沉默优于发声" — 不弹 toast. 但当 POST 同步失败时, 用户也只是回到 inbox 看到"未同步"小标记. 错误透明度可以再考虑.
OK ✓ 已正确实现的部分
字体 11 个文件全 bundle (Playfair · Source Serif 4 · Noto Serif SC · JetBrains Mono); 5 个 UI 组件 (Masthead/PaperCard/TapEffect/DoubleRule/SectionHeader) + Text 4 字号; theme 5 文件; ky 客户端 + Bearer 注入 + retry; TanStack QueryClient; B1 capture modal (无 toast/无 loading/无成功震动); A1 inbox FlatList + 下拉刷新 + 10s 轮询 + 本地/远端合并; SilenceStamp 今日沉默; signal/[id] 详情 + 推演结果展示; 包黑名单 (无 paper / elements / lottie / notifications) 干净.

§ IVLLM 端 (Mastra · /mastra)

推演链路完整可跑: NATS 消费 → Analyst Agent (Claude Sonnet 4.5) → schema 校验 → HTTP 回写后端. 主要缺口在 "质量验收" — 10 条 fixture 都在, 但跑完只统计"是否能 parse", 没有对照 rubric 自动打分.

严重度问题位置 / 影响
High manual eval 没有自动评分, "passed++" 只统计 parse 成功
mastra/tests/manual-eval/run.ts:41-67
M2 spec 要求 "10 条 ≥ 7 条产出符合预期". 当前 runner 只打印 JSON, 让人肉评分. 没有持久化打分文件, 没有 ≥7/10 判定, 没有 rubric 强制. Phase 1 的验收门很容易"过了又不算过".
High Analyst prompt 缺 few-shot 示例
mastra/src/agents/analyst.ts:18-42
prompt 解释了一/二/三阶概念和约束, 但没有任何 "输入 X → 期望输出 Y" 示例. Claude 偶尔会把 "供应商涨价" 推成 "供应商股价涨" — 这正是 prompt 显式禁止的, 但没示范.
Medium HTTP 工具无重试 / circuit breaker
mastra/src/tools/flashfi-api.ts (postInference 单次 fetch)
Go 临时 5xx → 整个 workflow 失败 → NATS nak 重投 → Analyst 重跑 (再付一次 token 钱). M2 spec Task 2.5 隐含 "persist 独立重试" 的设计意图. 当前实现把 LLM 调用也一起重跑了.
Medium env 校验在 module load, NATS 连接错误不 fast-fail
mastra/src/config/env.ts:8-14 + mastra/src/index.ts:8-9
ANTHROPIC_API_KEY 缺会立刻 throw, 但 NATS 连不上会进 reconnect 循环静默 log. 启动时应该明确 health-check, 否则部署时不知道 "其实 NATS 一直连不上".
Medium DLQ 只是 log + term, 没有真死信主题
mastra/src/consumers/nats.ts:113-125
3 次失败后 msg.term() 就丢了, 只在 stderr 留一行. Phase 1 spec 明确 "DLQ = 打日志报警" — 符合 spec, 但 Phase 2+ 必须有真 DLQ topic 或 dead_letter 表用于重放.
Medium 没有推演质量监控
整个 mastra/src/
线上推演没有: token 使用量, 推演置信度, 采样回看, 模型漂移检测. dur_ms 只是延迟. Phase 2 要加, Phase 1 没强制要求.
Low @ai-sdk/openai 装了但没用
mastra/package.json:20
Phase 1 显式选 Claude, OpenAI SDK 是死代码. 清理即可.
Low 日志靠 console.log + JSON.stringify, 无 logger
consumers/nats.ts, index.ts, 等
够用, 但 Phase 2 上量后要换 pino / winston, 否则改 log 格式要改每个 site.
Low 无单元 / 集成测试, 只有 manual eval
mastra/tests/ 只有 manual-eval/ 目录
workflow / consumer / tool 全无自动测试. 改 prompt 后没回归保护.
OK ✓ 已正确实现的部分
Analyst Agent (Claude Sonnet 4.5) + Flashfi 专属 prompt; InferenceSchema 在 generate() 时通过 output: zodSchema 强制约束 + 一次 retry; NATS JetStream consumer durable + manualAck + ackExplicit + deliverNew + maxDeliver(3) + ackWait 60s; signal-inference workflow analyze → persist; flashfi-api 内部 token header; 10 条 fixture (HBM / Mac Studio / DeepSeek / 电力 / 出口 / 咖啡 / GLP-1 / 航运 / 噪音 / 日本人才) 覆盖一/二/三阶 + 弱信号; env 强校验 (ANTHROPIC_API_KEY · NATS_URL · FLASHFI_API_URL · INTERNAL_TOKEN); 没有任何 Phase 2 agent 误埋.

§ V关闭 Phase 1 的最小工作清单

按"必须做才能 W8 自己用"排序. Critical 不补完, M4 验收必败.

  1. [Mobile · Critical] 装 expo-sqlite + drizzle-orm, 把 usePendingSignals 的 set/get 改成 SQL. schema 按 M4 spec §4.2 (含 syncStatus / serverEventId / attempts). ≈ 1.5 天
  2. [Mobile · Critical] 装 @react-native-community/netinfo, 在 PendingFlush 里订阅 isConnected, 重连时触发 retry. 加 AppState 监听 background→active 也触发. ≈ 1 天
  3. [Mobile · High] Sync Queue 加 mutex (防止并发 flush) + max 3 attempts (然后停, UI 显示 "未同步" 红字) + exponential backoff. ≈ 1 天
  4. [Mobile · Medium] Inbox 轮询改自适应: 有 pending 才 10s 轮, 全 done 后停. 配 refetchOnAppFocus. ≈ 0.5 天
  5. [LLM · High] manual-eval 输出 JSON 文件 + 半结构化打分 (每条 5 字段 pass/fail), 跑完打印 "7/10 passed" 真实判定. ≈ 0.5 天
  6. [LLM · High] Analyst prompt 加 3-5 个 few-shot 示例 (good 一阶 → 二阶推链, bad 瞎编示例). ≈ 0.5 天
  7. [Backend · High] 给 signal handler + service + outbox 写至少 6 个测试 (capture happy, capture dup, list pagination, get not found, RecordInference happy, internal auth 401). ≈ 1 天
  8. [Backend · Medium] Outbox 配置改 env 驱动 (OUTBOX_POLL_MS / OUTBOX_BATCH / OUTBOX_MAX_ATTEMPTS). ≈ 0.5 天
  9. [LLM · Medium] flashfi-api postInference 加 3 次重试 + 指数退避 (单独重试 HTTP, 不重跑 Analyst). ≈ 0.5 天
  10. [Mastra · Low] 删 @ai-sdk/openai 依赖. ≈ 5 分钟

总估时 ≈ 6.5 ~ 7 天 全职. 完成后即满足 M4 验收 (§验收标准 "用户行为" 全过), 可进入 W8 "自己用一周", 通过后才进 Phase 2 (M5 五轮追问).

3五层产品架构

信号 → 模式识别 → 确定性评估 → 承诺执行 → 复盘训练 (L1 → L5)

LAYER

信号捕捉 · 不让一线观察蒸发

"离信号最近, 离仓位最远" — 这条断链由此开始修
双通道: A 快速录入 (30 秒, 工作间隙一句话) · B 集中训练 (周 30 分钟, 五轮追问深挖) · C AI 自动后台推演 (异步)。

7架构亮点

这些设计值得保留, 它们就是产品哲学的物理形态

① events 表用 REVOKE 物理强制 append-only

不是应用层约定, 也不是触发器, 是 Postgres 在 query planner 阶段就拒绝 UPDATE/DELETE/TRUNCATE。TestEventsAreAppendOnly 集成测试是事件溯源的物理守门。
哲学映射: "让用户看见自己" 强制要求事件不可改 — 否则镜子会被涂改。

② LLM 准备数据 · 规则引擎做判断 · LLM 解释结果

四道门的判断永远在 Go 里 (G1/G3/G4 是确定性规则, G2 才接 LLM Consensus)。这避免了"LLM 全权决定"的失控风险。
哲学映射: "判据是产品良心" — 良心不能交给概率模型。

③ Narrator 字符级 verbatim 引用 + 三层兜底

承诺书的所有"用户原话"必须字符级匹配用户已说过的内容。Workflow 层做 substring 校验 → 不匹配 retry → 多次失败 NAK 让消息回队列。
哲学映射: "语言塑造行为" — 一旦 AI 给的"承诺"是它自己的话, 行为塑形失效。

④ M11.5 训练闭环

Diagnostician 复盘出"你这次最弱在推演力" → 写回 user_training_state.focus_dim → 下次 M5 五轮追问的 Socratic prompt 自动注入"重点测推演"。
哲学映射: 哲学 #7 "把性格工程化" 的物理实现。这是 Flashfi 跟"BI 报告"的分水岭。

⑤ 沉默归档 4 池, 不是"建议你看看"

四道门任一失败 = 系统沉默, 信号进观察池/教训池/日历池/舍弃池, 不弹任何 toast/push。
哲学映射: 哲学 #2 "沉默优于发声"。绝大多数 AI 产品做不到, 因为它们的 KPI 是 engagement。

⑥ Outbox + NATS JetStream + Durable Consumer

事务内 events 表 + outbox 表一起 insert, 异步发布; durable consumer 保证 at-least-once。
哲学映射: 录入永不阻塞 (用户 30 秒内一定能继续工作), 但事件不会丢。

8完善建议

按优先级排, 每条都先说"为什么", 再说"做什么", 最后说"边界"

A · 必做 (W8 自己用之前)

A1 · 把质量层从 35% 推到至少 70%
最高优先
Why · 代码层 100% 是假象 — Phase 2/3 backend handler 测试 0%, Mastra eval fixtures 只有 Analyst (10条)。自己用一周第一天就可能撞 production 缺陷。
What:
  • 给 M5/M6/M7/M8/M9/M11 各补 3-5 个 handler 集成测试 (按 M2 的 handler_test.go 模板)
  • Socratic / Narrator / Diagnostician 各写 10 条 eval fixtures + 自动评分 (verbatim 校验 是必查项)
  • 跑通整套 make test 在 .205 服务器, 把命令塞进 remote-sync.sh 的 post-build 步骤
边界: 不写 e2e UI 测试 (Phase 1 没人维护 e2e fixtures); 不引 Crashlytics/Sentry (违反 TestFlight § 9)。
A2 · 补上 audit 里那条 PARTIAL: GET /v1/signals 按 status 过滤
必修
Why · Mobile 客户端目前是全量拉信号, 用一周后信号变多, 启动 < 2 秒的承诺会破。这是 M4 唯一 PARTIAL 项。
What:
  • signal handler 加 ?status=accepted|archived|all query 支持
  • 客户端 inbox 默认拉 accepted; archive tab 拉 archived
  • handler 测试加一例: 100 条信号 + status=accepted 只回 N 条
边界: 不引入分页 (Phase 1 单用户 7 天数据量不需要); 不加 cursor (留给 Phase 3)。
A3 · LLM provider fallback 透明化
高
Why · 6 个 Agent 全接 Claude / DeepSeek。一旦 LLM 挂了或限流, 当前是"自动 fallback 到启发式"。用户看到的不是"AI 在工作", 是"AI 出错了但你不知道"。这会破坏哲学 #2 "沉默优于发声" — 因为这时候说的话不是真的 AI 判断。
What:
  • 每个 Agent 加 llm_status: "real" | "fallback" 字段写进 events 表
  • web-admin Metrics 页加"LLM 健康度"卡 — 过去 24h fallback 率, >5% 报警
  • 当 fallback 触发时, mobile 不要展示推演结果 — 改为沉默归档 (一致性)
边界: 不引第二个 provider 做 hot-failover (复杂度爆炸); 沉默是默认。

B · 该做 (W8 自己用过程中持续观察)

B1 · 行为指纹假阳性的人工标注闭环
中高
Why · 风险登记 #4 — "行为指纹假阳性 → 焦虑卡反而制造焦虑"。M9 完成代码不等于完成判据, 必须 10 天人工标注。这是 Phase 3 最大风险, 现在没有标注工具。
What:
  • web-admin 加一个 /companion-labeling 页, 列出过去 7 天所有 fingerprint 触发的 editor 调用
  • 每条加 [真焦虑 / 假阳性 / 没意义] 三按钮, 写回 fingerprint_labels 表 (新 migration)
  • 记录 10 天, 如果假阳性 > 30%, 降低触发率而非调阈值 (按风险登记 #4 的缓解方案)
边界: 单人手工标 (Phase 1 单用户); 不引入 LLM 自动标 (会污染判据)。
B2 · 让"自己用一周"有可见的物理证据
中高
Why · 风险 #1 是致命: "半年后失去兴趣"。当前"自己用一周"靠诚信 — 没有 ⬜→✅ 的物理化判定。一个客观计数器会变成强烈的自我提醒。
What:
  • 在 web-admin Dashboard 加"连续录入天数"指标 + 当周录入热力图 (类似 GitHub 草坪)
  • 缺一天 → 醒目红字"自己用一周 · 中断" — 但不发任何通知 (沉默)
  • Phase 1 W8 / Phase 2 W18 / Phase 3 W26 三个里程碑分别绑定到这个计数器
边界: 这不是"streak gamification" — 没奖励, 没分享, 只是镜子; 中断后不"撤销", 历史诚实保留。
B3 · 给 events 表加 redact_user(uuid) SECURITY DEFINER 函数 (前置)
中
Why · ADR 0002 已经写明: "如果合规要求删用户数据, 不开 DELETE, 而是写 SECURITY DEFINER 函数把 payload 替换成 redacted, 保留 id/type/时间戳"。TestFlight 进来 5 个内测者就要面对这个问题 — 谁要离场, 怎么处理?
What:
  • 新 migration 009 加 redact_user(uuid) 函数
  • handler 集成测试: 调用后再 SELECT, 看 id/type/created_at 还在, payload 变成 {"redacted":true}
  • web-admin Settings 页加"导出/抹除我的数据"按钮 (单用户场景也要有, 它本身是哲学 #10 "产品消失" 的物理形态)
边界: 不引 GDPR 全套 (单用户不需要); 不删派生表 (重建即可)。
B4 · 把 docker-compose 加上 restart: unless-stopped + 启动自检
中
Why · SERVER.md 灾难恢复一节已经标注: "服务器重启后 Docker 容器没设 restart, 暂时手动 docker compose up -d"。这是已知坑, 一次断电就毁掉数据底座的 SLA。
What:
  • docker-compose.yml 两个 service 都加 restart: unless-stopped
  • flashfi-api systemd unit 加 After=docker.service + Requires=docker.service
  • healthz 加深度: 不仅返回 ok, 还检查 postgres + nats 都可达
边界: 不上 k8s; 不引 watchdog (1 台机器单用户)。

C · 可做 (Phase 1 之后再评估)

C1 · "想法 1" 那条线 (项目目标 → 关键里程碑 → 自动拆任务)
低
Why · 总览文档明确说产品同时承载"投资信号"和"项目目标"两条线, 底层是同一件事。当前代码 100% 是"投资信号" — "项目目标"那条线一行未动。但这是大改, 不该在 W8 前碰。
What (Phase 4 候选):
  • 评估 events 表 + 物化视图能否承载 milestones / tasks (大概率可以, 它本就是 polymorphic)
  • 评估 M5 五轮追问能否复用到"目标拆解" (锚定预期/现实/意外 → 锚定北极星/瓶颈/取舍)
  • 评估 M7 承诺书能否复用 (signing 不变, payload schema 变)
边界: W26 v1.0 之前不做。两条线一起搞会让验收变浑浊, 失去"自己用一周"的判定能力。
C2 · Mobile 加触觉反馈 + 字体细节 (native_feel_skill 落地)
低
Why · 技术文档/native_feel_skill 有完整 SKILL + 8 references + 2 checklists, 当前 core/haptics.ts 已经铺位但还没在所有交互点接好。"产品形态是对话式收件箱, 默认安静" — 安静的产品对触感和字距更敏感。
What:
  • 按 native_feel_skill checklist 过一遍现有屏: 签字按钮 medium impact, 录入提交 light selection, 焦虑卡展开 selection
  • 字体: 标题 SF Pro Display, 正文 SF Pro Text, 数字等宽 (承诺书的仓位/价格)
  • Safe area / Dynamic Island 兼容
边界: 不引 Lottie 动画; 不做主题切换; 单用户不需要 i18n。
C3 · pgvector 真正用起来 (Phase 3+)
低
Why · docker-compose 用的是 pgvector/pgvector:pg16 镜像, 但当前 schema 没用 vector 列。复盘训练的"找相似历史案例"是 pgvector 的本职 — "你这次和 2024 年 1 月那次类似, 当时你也卡在门 4"。
What (Phase 3 末):
  • 给 signals 加 embedding 列 (raw_text 入库后异步算)
  • Diagnostician 多一个 tool: find_similar_past_cases(current_signal_id)
  • 复盘卡多一个 module: "和你过去 K 个案例的相似度", 让用户看见自己的模式
边界: K ≤ 3, 不要让模式识别变成推荐系统; embedding 只在本机推理 (绝不走外部 API)。
C4 · 产品消失的物理形态: 数据导出
低
Why · 哲学 #10 "最高境界是产品消失"。当用户三个月后把五轮追问内化成思维习惯, 他/她应该能带着所有数据离开 — 这是产品自洽的最后一环。
What (W26 v1.0 之前):
  • web-admin 加 /export: 全部 events + 派生视图打包 ndjson
  • 导出包含一份 README, 说明 events 表的 schema (让用户用 jq 也能查)
  • 导出按钮旁边写一句话: "你的所有判断都在这里。任何时候你可以带走它们。"
边界: 不做"导入回来"功能 (产品消失就是消失); 不做云同步 (单设备单用户)。

# M6 · 四道门评估引擎

> Phase 2 · W12-W13 · 2 周 · 规则引擎为主, LLM 为辅

---

## 上下文

M6 是产品的**良心所在**。

四道门决定一条信号能不能进入"承诺书"阶段。
任一门失败 → **沉默归档**, 不降级提示。

这是产品哲学"沉默优于发声"的物理实现。

---

## 前置依赖

- ✅ M5 五轮追问完成(评估依赖追问产出的数据)

---

## 目标

### 规则引擎(Go 实现, 不是 LLM)
- 门 1: 信号厚度(独立信号 ≥ 3 条)
- 门 2: 反共识程度(对比市场情绪指标)
- 门 3: 时间窗口(1-6 月)
- 门 4: 个人认知优势(4 个子测试)

### 归档系统
四个池:
- 门 1 失败 → 观察池(继续累积)
- 门 2 失败 → 教训池
- 门 3 失败 → 日历池(等窗口)
- 门 4 失败 → 舍弃池

### 客户端展示
- A1 收件箱里的归档池入口(隐藏在"档案" tab)
- 不在主流程显示(沉默是默认)

## 任务列表(高层)

### Task 6.1 · 门 1 信号厚度 · 规则实现

Go 代码, 不依赖 LLM:

```go
func evaluateGate1(signals []Signal) GateResult {
    // 找到指向同一资产/主题的独立信号
    // 独立 = 不同来源(同事 / 群 / 客户 / 自己观察)
    // ≥ 3 条独立 → PASS
}
```

数据来源: 信号的 `inference.related_assets` + `inference.tags`。

**已知坑**:
- "独立" 的判断是难点, M5 已经在追问中收集了 source 信息, 用它
- 简化版: 用 inference.tags 的重合度 + 时间窗口判断

### Task 6.2 · 门 2 反共识 · LLM + 规则混合

LLM 部分: 查询当前主流研报覆盖度(用 web search tool 或预设数据集)。
规则部分: 价格走势 vs 主流叙事的偏离度。

Phase 2 简化: **LLM 给一个 0-100 的"主流热度"分数**, 70 以下算反共识。

### Task 6.3 · 门 3 时间窗口 · 纯规则

```go
func evaluateGate3(thesis Thesis) GateResult {
    window := thesis.WindowMonths
    if window < 1 || window > 6 {
        return Fail("窗口不合适")
    }
    return Pass()
}
```

窗口从 M5 追问的第 5 轮里来(用户自己估的时间)。

### Task 6.4 · 门 4 认知优势 · 4 个子测试

最严格的一关, 4 个子测试全过才算门 4 通过:

1. **能用一句话向不懂行业的朋友解释吗?** → 用户在 M5 追问里答的
2. **是直接接触的信号吗?** → 检查 M5 追问的 source 字段
3. **过去 12 个月在这个领域有过成功预判吗?** → 查历史 events 表
4. **能说出失败条件吗?** → M5 第 5 轮强制收集

**已知坑**:
- 第 3 项 Phase 2 简化: 默认 false(冷启动用户), Phase 3 复盘机制完善后才能用
- 第 4 项是关键, M5 必须收集这个数据

### Task 6.5 · 沉默归档逻辑

```go
func archiveSilently(signal Signal, failedGate int) {
    switch failedGate {
    case 1: moveToPool("observation", signal)
    case 2: moveToPool("lesson", signal)
    case 3: moveToPool("calendar", signal)
    case 4: moveToPool("discard", signal)
    }
    // 不通知客户端, 不写 push 事件
}
```

**最重要**: 失败时不弹任何提示, 不让用户感知到"四道门评估发生过"。
用户只在主动打开"档案"页时才看到归档池。

### Task 6.6 · 全过的提示

四门全过 → 写 events 表 `gate.passed`, 客户端在下次打开时 A1 顶部看到一张特殊卡:

> "我从你最近的几条信号里看到一个可能值得下注的事, 要不要展开聊聊。"

**注意**: 这张卡也不是 push, 是用户打开 APP 时看到的。

### Task 6.7 · 档案 Tab 实现

`app/(tabs)/archive.tsx`:
- 顶部: 4 个池的入口(图标 + 数量)
- 进入某个池: 看到所有归档信号的列表
- 每条显示: 当时为什么没过(门 N 失败的原因)

## 反模式特别警告

- ❌ 不要"降级提示"("虽然门 2 没过, 但可以考虑")
- ❌ 不要给"分数"("门 1 得 75/100")
- ❌ 不要可视化为漏斗 / 流程图
- ❌ 不要弹"你的信号未通过评估"的通知

正确形态:**用户大多数时候不知道四道门存在**。只有"四门全过"是被看到的。

## 已知坑

1. **门 4 第 3 项冷启动是 false**, Phase 3 后才有数据填这一项
2. **沉默归档不写 NATS 消息**, 不要让客户端订阅到
3. **池的更新用物化视图**, 不要每次扫 events
4. **门 1 的"独立信号"判断模糊**, 用 tags 重合 + 时间窗口启发式
5. **门 2 LLM 评估不稳定**, 多跑几次取众数, 或加缓存

§ 01 · THE QUESTION

从物理学开始

之前的团队问的是"我们想做什么产品"。第一性原理要求我们问"什么是物理上必须存在的"。 我读了 06_产品哲学.md, 写得很美。但美是结果, 不是出发点。

Q1 · 客户的根本需求是什么
从一线观察到资本配置, 这条链路平均损耗 99%。每年只发生 1-2 次的高确定性机会, 大多数人识别了, 然后蒸发了。
→ 物理量: 信号到仓位的转化率。不是"engagement", 不是"DAU", 是 conversion。当前业内是 ~0.5%。可以推到 30%+。
Q2 · alpha 的物理本质
不是信息差。是从认知到行动的延迟差。两个人看到同样的供应商抱怨, 一个人 6 个月后回看说"我当时就知道", 另一个人下注了。差距不在信号, 在物理上的执行链路。
→ 物理量: signal-to-position latency。当前: 6-12 个月 (从想到到下单, 如果有的话)。目标: 72 小时。
Q3 · 为什么"匹配 alpha 频率"是错的
前团队说"一年 1-2 次主动提示"。这把因果搞反了。 alpha 是稀有事件 — 没错。但识别它的过程不是稀有的。识别需要每天的训练。产品要做的不是变稀有, 是把训练做密, 让稀有事件来时, 用户的判断力是 ready 的。
→ 日常训练 ≠ 日常打扰。日常训练是用户主动来; 打扰是产品推过去。前团队把两件事混淆了。
Q4 · 单用户假设的代价
26 周 / 11 个模块 / 6 个 Mastra agent — 全部假设 N=1。这意味着训练数据永远只有一个人的样本。一个人的复盘训练永远不可能比 1000 个人的复盘训练更准。Diagnostician 在 N=1 下永远是过拟合。
→ 把 N 从 1 改成 ∞ 是这次接管的核心动作。其他都是推论。

§ 03 · THE ALGORITHM

五步法 · 顺序不能颠倒

我在 Tesla / SpaceX 用了 20 年。这五步不是建议, 是顺序。颠倒任何一步就会过早优化或保留废物。

1

Question every requirement

质疑每一个需求 · 包括产品哲学里的

"默认沉默"是哲学。但对谁沉默从未问过。对当前用户沉默 ≠ 对全体用户沉默。 一个用户的"我没识别到这条信号"在 N=1000 时可能是另一个用户的训练靶子。 "26 周路线图"也要质疑 — 是因为审慎吗, 还是因为我们害怕被验证错了?

→ 重写 GOAL.md 第一段。
→ 26 周计划砍掉所有"等下个 Phase"约束。
2

Delete the part / process

最常被跳过的一步 · 删错了再加回来比从不删要好 10 倍

前团队的所有反思都在"加什么"。我看到 4 个东西可以立刻删 (见 § 04)。 如果我们删错了一个, 1 周内会发现它的缺失 — 那时候再加回来。 如果什么都没删, 说明你还没真用过这个算法。

→ web-admin / Mastra 中间层 / 11 个模块 / 26 周 phase gate — 都进 delete review。
3

Simplify and optimize

这一步只有在删过之后做才有意义

如果先优化再删, 你会优化一些根本不该存在的东西。前团队在 verbatim 校验上花的工程量 (workflow substring + retry + NAK 三层兜底) — 这是对一个不该存在的复杂度的精致防御。 真正的简化是: 让 LLM 用 user 原话生成时, 用 grammar-constrained decoding, 物理上让它不能生成原话之外的字。

→ 评估 Outlines / llama.cpp grammar 替换三层兜底。一周原型。
4

Accelerate cycle time

在删 / 简化之后, 不在之前

ship 频率从"每个 Phase 自己用一周" 改成 "every weekday end-of-day"。 mobile app 每天有一个新 TestFlight build (EAS 已经配好, 跑就是了)。 backend 每个 commit 自动 deploy 到 .205 (remote-sync.sh 已经差不多了 — 加一个 GitHub Actions 触发)。

→ 砍掉 W8/W18/W26 三个"自己用一周" gate。改成 every Friday demo, 不通过 rollback。
5

Automate

最后一步 · 自动化错的东西比手动做错的东西贵 10 倍

目前可自动化但没做的事: ①Mastra eval fixtures 跑 CI (现在手动) ②mobile 在 expo update channel 上自动分发 (现在手动 EAS submit) ③复盘训练的 4 问让 LLM 主动从持仓动作里提问, 不再用户填表。 但: 自动化"找 alpha 的判断"是禁区 — LLM 全权决定哪些信号过门, 等于放弃产品良心 (前团队这一点是对的, 保留)。

→ CI/CD pipeline · expo update channel · LLM 主动复盘问询。三件事 4 周内完成。

VERBATIM HARDENING

  • 本地跑一个开源模型 (Llama 3.1 70B 或 Qwen 2.5 72B) 处理 verbatim 任务

  • 这一改使得 verbatim 错误率从 ~5% (retry 后) → 0% (物理上)
  • 同时降低延迟 (省一次 retry) + 成本 (本地模型 vs Claude)

§ 09 · WHAT IT BECOMES

这不是
投资工具。

它是一台 把"差一点就买了"转化为"已经买入并持有 180 天"的物理设备。

它的存在不为了让任何人"赚钱"。 它的存在是为了 让人类对自己的判断力的迭代速度, 提高一个数量级。

v0 团队把这个产品做成了一份美学品。
v1 我把它做成一台 引擎。

美学品 vs 引擎的区别:
